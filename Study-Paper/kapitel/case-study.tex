\section{Case Study: Umsetzung eines vollständigen Interpreters in TypeScript}
\label{sec:case_study}

In diesem Kapitel wird die praktische Umsetzung der zuvor entwickelten Konzepte anhand des letzten Notebooks (03-Interpreter.ipynb) aus Chapter-07 zur Interpreter-Realisierung dargestellt. Im Mittelpunkt steht nicht nur ein einzelner Ausdrucksparser, sondern eine vollständige Verarbeitungskette für eine kleine \texttt{C}-ähnliche Sprache: von der Grammatikdefinition über die Parsergenerierung und die Transformation des Concrete Syntax Tree (CST) in einen Abstract Syntax Tree (AST) bis hin zur eigentlichen Interpretation des Programms.

Die Fallstudie ist besonders geeignet, um die Unterschiede und Gemeinsamkeiten zwischen der ursprünglichen Python-basierten Lehrumgebung und der TypeScript-Variante sichtbar zu machen. Beide Implementierungen folgen denselben compilertechnischen Grundprinzipien, unterscheiden sich jedoch in der Modellierung der Datenstrukturen, im Zusammenspiel mit dem Typsystem und in der konkreten Werkzeugkette. Der grundsätzliche Ablauf entspricht den klassischen Phasen eines Sprachprozessors: syntaktische Analyse, interne Repräsentation und semantische Ausführung \autocite[S.~4\,ff.]{aho2006compilers}.

\subsection{Ziel und Aufbau des Notebooks}

Das Notebook implementiert einen vollständigen Interpreter für eine kleine imperative Sprache mit Variablen, arithmetischen Ausdrücken, Bedingungen, Schleifen und elementaren Funktionsaufrufen. Die Verarbeitungskette besteht aus vier Schritten:

\begin{enumerate}
    \item Definition der Sprache durch eine formale Grammatik,
    \item Generierung eines Parsers mit Lezer,
    \item Umwandlung des erzeugten CST in einen kompakteren AST,
    \item rekursive Interpretation des AST.
\end{enumerate}

Diese Struktur entspricht sowohl in der Python- als auch in der TypeScript-Implementierung dem klassischen Aufbau eines Interpreters. Während die Python-Variante typischerweise auf PLY basiert und Lexer sowie Parser explizit trennt, wird in der TypeScript-Version mit Lezer eine integrierte, deklarative Grammatik verwendet. Unabhängig vom eingesetzten Werkzeug bleibt die zugrunde liegende Architektur identisch. Eine formale Grammatik beschreibt die syntaktisch zulässigen Programme, der Parser konstruiert daraus eine Baumdarstellung, und ein nachgelagerter Verarbeitungsschritt interpretiert diese Repräsentation semantisch \autocite[S.42ff.]{aho2006compilers}.

\subsection{Sprachdefinition im Vergleich: Python (PLY) vs. TypeScript (Lezer)}

In der ursprünglichen Python-Implementierung wird die Sprache mithilfe von \emph{PLY} (Python Lex-Yacc) definiert. Dabei erfolgt eine klassische Trennung zwischen lexikalischer und syntaktischer Analyse, wie sie in der Compilertechnik etabliert ist \autocite[S.~5ff]{aho2006compilers}. Die Tokenisierung wird durch reguläre Ausdrücke beschrieben, während die Grammatikregeln in Form von Python-Funktionen implementiert werden. Dieses Vorgehen orientiert sich eng an traditionellen Werkzeugen wie \emph{Lex} und \emph{Yacc} und spiegelt den zweiphasigen Aufbau eines Compilers wieder.

\begin{listing}[htbp]
\caption{Ply-Grammatik}
\label{lst:ply-grammatik}
\begin{minted}{python}
import ply.lex as lex
tokens = [ 
    'NUMBER',     # r'0|[1-9][0-9]*'
    'IDENTIFIER', # r'[a-zA-Z][a-zA-Z0-9_]*'
    'ASSIGN',     # r':='
    'EQ',         # r'=='
    'NE',         # r'!='
    'LE',         # r'<='
    'GE',         # r'>='
    'IF',         # see below
    'WHILE'       # see below
]

def t_COMMENT(t):
    r'/\*(.|\n)*?\*/|//.*'
    t.lexer.lineno += t.value.count('\n')

def t_NUMBER(t):
    r'0|[1-9][0-9]*'
    t.value = int(t.value)
    return t

t_ASSIGN = r':='
t_EQ     = r'=='
t_NE     = r'!='
t_LE     = r'<='
t_GE     = r'>='

Keywords = { 'if': 'IF', 'while': 'WHILE' }

def t_IDENTIFIER(t):
    r'[a-zA-Z][a-zA-Z0-9_]*'
    t.type = Keywords.get(t.value, 'IDENTIFIER')
    return t

literals = ['+', '-', '*', '/', '%', '(', ')', '{', '}', ';', '<', '>', ',']

t_ignore  = ' \t\r'

def t_newline(t):
    r'\n'
    t.lexer.lineno += 1
    return


def find_column(token):
    program    = token.lexer.lexdata  # the complete string given to the scanner
    line_start = program.rfind('\n', 0, token.lexpos)
    return token.lexpos - line_start

def t_error(t):
    column = find_column(t)
    print(f"Illegal character '{t.value[0]}' in line {t.lineno}, column {column}.")
    t.lexer.skip(1)

lexer = lex.lex()  
\end{minted}
\end{listing}

Dieses Vorgehen führt zu einer engen Verzahnung von Grammatik und Programmlogik. Grammatikregeln sind implizit in Python-Code eingebettet, wodurch die formale Struktur der Sprache weniger explizit sichtbar ist. Gleichzeitig ermöglicht dieser Ansatz eine flexible Integration semantischer Aktionen direkt in den Parser.

Im Gegensatz dazu wird die Sprache in der TypeScript-Implementierung mit \emph{Lezer} deklarativ als Grammatik spezifiziert. Die Grammatik wird dabei als strukturierter String formuliert und anschließend automatisch in einen Parser übersetzt. Dieses Vorgehen entspricht stärker der formalen Definition kontextfreier Grammatiken, wie sie in der Theorie formaler Sprachen üblich sind \autocite[S.~171ff]{hopcroft2006automata}.

\begin{listing}[htbp]
\caption{lezer-Grammatik}
\label{lst:lezer-grammatik}
\begin{minted}{typescript}
import { buildParser } from "@lezer/generator";
const grammarString = `
    @top Script { Program }

    @tokens {
        Identifier { $[a-zA-Z] $[a-zA-Z0-9_]* }
        Number     { "0" | $[1-9] $[0-9]* }

        "+" "-" "*" "/" "%"
        ":=" "==" "!=" "<=" ">=" "<" ">"
        "(" ")" "{" "}"
        ";" "," 

        space { $[ \t\n\r]+ }
        LineComment { "//" ![\n]* }
        BlockComment { "/*" ( ![*] | "*" + ![*/] )* "*"+ "/" }
        @precedence { LineComment, BlockComment, "/" }
    }

    @skip { space | LineComment | BlockComment }

    IF    { @specialize<Identifier, "if"> }
    WHILE { @specialize<Identifier, "while"> }

    Program {
        Stmnt Program | 
        ""
    }

    Stmnt {
        IF "(" BoolExpr ")" Stmnt    |
        WHILE "(" BoolExpr ")" Stmnt |
        "{" Program "}"              |
        Identifier ":=" Expr ";"     |
        Expr ";"
    }

    BoolExpr {
        Expr "==" Expr |
        Expr "!=" Expr |
        Expr "<=" Expr |
        Expr ">=" Expr |
        Expr "<"  Expr |
        Expr ">"  Expr
    }

    Expr {
        Expr "+" Product |
        Expr "-" Product |
        Product
    }

    Product {
        Product "*" Factor |
        Product "/" Factor | 
        Product "%" Factor |
        Factor
    }

    Factor {
        "(" Expr ")"                |
        Number                      |
        Identifier                  |
        Identifier "(" ExprList ")"
    }

    ExprList {
        NeExprList | 
        ""
    }

    NeExprList {
        Expr |
        Expr "," NeExprList
    }
`;
const parser = buildParser(grammarString);
\end{minted}
\end{listing}

Ein wesentlicher Unterschied zwischen beiden Ansätzen liegt somit im Abstraktionsniveau: Während PLY eine imperativ definierte Kombination aus Lexer- und Parserlogik darstellt, beschreibt Lezer die Sprache deklarativ auf Basis einer expliziten Grammatik. Diese deklarative Form erleichtert insbesondere die Analyse, Wartbarkeit und Weiterverarbeitung der Grammatik, da die Struktur der Sprache klar getrennt von ihrer Verarbeitung beschrieben wird.

Darüber hinaus unterstützt Lezer moderne Anforderungen wie inkrementelles Parsing, das insbesondere für editornahe Anwendungen relevant ist. Der Parser kann dabei Änderungen effizient verarbeiten, ohne die gesamte Eingabe neu analysieren zu müssen \autocite[S.~3ff]{grune2008parsing}. Dies stellt einen wesentlichen Vorteil gegenüber klassischen Parsergeneratoren dar, die typischerweise eine vollständige Neuverarbeitung der Eingabe voraussetzen.

Zusammenfassend lässt sich festhalten, dass PLY einen eher traditionellen, eng an der Implementierung orientierten Ansatz verfolgt, während Lezer eine stärker formal orientierte und deklarative Beschreibung der Sprache ermöglicht. Diese Unterschiede sind insbesondere im Kontext der Übersetzung von Python nach TypeScript relevant, da sie unterschiedliche Möglichkeiten zur Modellierung und Weiterverarbeitung von Sprachstrukturen bieten.

\subsection{Transformation vom CST zum AST}

Ein wesentlicher Unterschied zwischen beiden Implementierungen zeigt sich unmittelbar nach dem Parsing. In der Python-Version wird der AST bereits während der Anwendung der Grammatikregeln erzeugt. Die semantischen Aktionen der PLY-Produktionen konstruieren direkt die interne Repräsentation, sodass keine separate Transformationsphase zwischen Parsebaum und abstrakter Syntax mehr erforderlich ist. Die Grammatik und die Konstruktion der Zielstruktur sind damit eng miteinander verbunden.

\begin{listing}[htbp]
\caption{Direkte AST-Konstruktion in PLY}
\label{lst:ply-ast-direkt}
\begin{minted}{python}
def p_stmnt_if(p):
    "stmnt : IF '(' bool_expr ')' stmnt"
    p[0] = ('if', p[3], p[5])   

def p_stmnt_while(p):
    "stmnt : WHILE '(' bool_expr ')' stmnt"
    p[0] = ('while', p[3], p[5])
    
def p_stmnt_block(p):
    "stmnt : '{' program '}'"
    p[0] = p[2]
    
def p_stmnt_assign(p):
    "stmnt : IDENTIFIER ASSIGN expr ';'"
    p[0] = (':=', p[1], p[3]) 

def p_stmnt_expr(p):
    "stmnt : expr ';'"
    p[0] = ('expr', p[1])
\end{minted}
\end{listing}

Die TypeScript-Variante trennt diese Schritte klarer. Lezer erzeugt zunächst einen CST, der anschließend durch eine eigene Transformationsfunktion in einen AST überführt wird. Dadurch wird die klassische Trennung zwischen konkreter und abstrakter Syntax explizit sichtbar \autocite[S.~42ff.]{aho2006compilers}.

\begin{listing}[htbp]
\caption{CST-zu-AST-Transformation in TypeScript}
\label{lst:ts-cst-zu-ast}
\begin{minted}{typescript}
import { cst2ast } from './CST2AST';

const myOperators = [
    "IF", "WHILE", ":=",              
    "+", "-", "*", "/", "%",
    "==", "!=", "<", ">", "<=", ">="
];

const myListVars = ["Program", "NeExprList"];

function parse(fileName: string): AST {
    const source = readFileSync(fileName, "utf8");
    const tree   = parser.parse(source);
    return cst2ast(tree.cursor(), source, myOperators, myListVars);
}
\end{minted}
\end{listing}

Die Python-Variante erzeugt ihre Zielstruktur somit unmittelbar innerhalb der Grammatikregeln, während die TypeScript-Version einen expliziten Zwischenschritt einführt. Diese zusätzliche Transformationsphase erhöht die Modularität der Implementierung und trennt die formale Beschreibung der Sprache deutlicher von ihrer internen Repräsentation.

\subsection{Darstellung rekursiver Listenstrukturen}

Auch bei der Modellierung rekursiver Listenstrukturen folgen beide Notebooks demselben Grundprinzip. Sowohl Programmlisten als auch Argumentlisten werden als Punktlisten dargestellt. In der Python-Version geschieht dies direkt in den PLY-Regeln.

\begin{listing}[htbp]
\caption{Rekursive Listen in der Python-Version}
\label{lst:ply-listen}
\begin{minted}{python}
def p_program_lambda(p):
    "program : "
    p[0] = ('.',)

def p_program_more(p):
    "program : stmnt program"
    p[0] = ('.', p[1]) + p[2][1:]

def p_expr_list_empty(p):
    "expr_list : "
    p[0] = ('.',)
     
def p_expr_more(p):
    "expr_list : ne_expr_list"
    p[0] = p[1]     

def p_ne_expr_list_one(p):
    "ne_expr_list : expr"
    p[0] = ('.', p[1]) 
    
def p_ne_expr_list_more(p):
    "ne_expr_list : expr ',' ne_expr_list"
    p[0] = ('.', p[1]) + p[3][1:]
\end{minted}
\end{listing}

In der TypeScript-Version bleibt diese Listenstruktur ebenfalls erhalten, wird jedoch explizit durch eine eigene Ausführungsfunktion verarbeitet. Die rekursive Form wird damit nicht nur beim Aufbau des AST, sondern auch bei seiner semantischen Verarbeitung sichtbar.

\begin{listing}[htbp]
\caption{Verarbeitung rekursiver Listen in TypeScript}
\label{lst:ts-listen}
\begin{minted}{typescript}
executeList = (list: AST, env: Environment) => {
    if (list === "") return;
    
    if (Array.isArray(list) && list[0] === '.') {
        const head = list[1];
        const tail = list[2];

        if (head !== undefined && tail !== undefined) {
            execute(head, env);
            executeList(tail, env);
        }
    }
};
\end{minted}
\end{listing}

Die Gemeinsamkeit beider Varianten liegt darin, dass die rekursive Struktur der Grammatik unmittelbar in die interne Repräsentation übernommen wird. Der Unterschied besteht vor allem darin, dass TypeScript diese Listenstruktur zusätzlich durch Hilfsfunktionen systematischer kapselt.

\subsection{Typisierung und interne AST-Repräsentation}

Ein besonders markanter Unterschied zwischen beiden Implementierungen liegt in der Darstellung des AST. In Python bleibt die Baumstruktur implizit durch Tupelkonventionen beschrieben. Welche Form ein Knoten besitzt, ergibt sich aus dem verwendeten Tag und der Position seiner Operanden. In TypeScript wird dieselbe Struktur dagegen durch einen expliziten Typ beschrieben.

\begin{listing}[htbp]
\caption{AST-Typdefinition in TypeScript}
\label{lst:ts-ast-typ}
\begin{minted}{typescript}
type Operator = string;
type AST = string | number | 
           [Operator, AST]                |
           [Operator, AST, AST]           |
           [Operator, AST, AST, AST]      |
           [Operator, AST, AST, AST, AST];
\end{minted}
\end{listing}

Diese Typdefinition dokumentiert die rekursive Struktur des AST unmittelbar im Programmtext. Während in Python nur informell klar ist, dass etwa \texttt{('+', lhs, rhs)} einen binären Ausdruck repräsentiert, wird in TypeScript explizit festgelegt, dass AST-Knoten aus Zahlen, Bezeichnern oder Operator-Tupeln verschiedener Stelligkeit bestehen. Dadurch wird die Baumstruktur besser dokumentiert und teilweise statisch überprüfbar.

\subsection{Interpreterarchitektur im Vergleich}

Die Grundstruktur des Interpreters bleibt in beiden Notebooks gleich. Sowohl in Python als auch in TypeScript erfolgt eine Trennung in Anweisungsausführung, arithmetische Auswertung und boolesche Auswertung. In der Python-Version wird dies durch Pattern Matching über Tupel realisiert.

\begin{listing}[htbp]
\caption{Ausführung von Anweisungen in Python}
\label{lst:ply-execute}
\begin{minted}{python}
def execute(stmnt: NestedTuple, Values: dict[str, Number]) -> None:
    match stmnt:
        case ('.', *SL):
            execute_tuple(tuple(SL), Values)
        case (':=', var, value):
            Values[var] = evaluate(value, Values)
        case ('expr', expr):
            evaluate(expr, Values)
        case ('if', test, stmnt):
            if evaluate_bool(test, Values):
                execute(stmnt, Values)
        case ('while', test, stmnt):
            while evaluate_bool(test, Values):
                execute(stmnt, Values)
        case _:
            assert False, f'{stmnt} unexpected'
\end{minted}
\end{listing}

Die TypeScript-Version folgt derselben semantischen Idee, verwendet aber explizite Fallunterscheidungen über Knotentags.

\begin{listing}[htbp]
\caption{Ausführung von Anweisungen in TypeScript}
\label{lst:ts-execute}
\begin{minted}{typescript}
execute = (node: AST, env: Environment): void => {    
    if (!Array.isArray(node)) return;
    const tag = node[0];

    if (tag === '.') {
        executeList(node, env);
        return;
    }

    if (tag === "block") {
        const content = node[1];
        if (content !== undefined) {
            executeList(content, env);
        }
    }
    else if (tag === ":=") {
        const name = node[1];
        const expr = node[2];
        if (typeof name === "string" && expr !== undefined) {
            env.set(name, evaluate(expr, env));
        }
    }
    else if (tag === "IF") {
        const cond = node[1];
        const body = node[2];
        if (cond !== undefined && body !== undefined) {
            if (evaluateBool(cond, env)) execute(body, env);
        }
    }
    else if (tag === "WHILE") {
        const cond = node[1];
        const body = node[2];
        if (cond !== undefined && body !== undefined) {
            while (evaluateBool(cond, env)) execute(body, env);
        }
    }
    else if (tag === "Call") {
        evaluate(node, env);
    }
};
\end{minted}
\end{listing}

Die semantische Struktur bleibt also erhalten, obwohl sich die sprachspezifischen Mittel unterscheiden. Python nutzt Tupel und Pattern Matching, TypeScript verwendet Array-Prüfungen und String-Tags.

\subsection{Auswertung arithmetischer Ausdrücke}

Auch bei der Auswertung arithmetischer Ausdrücke zeigen sich deutliche strukturelle Parallelen. In Python wird die Semantik direkt durch rekursive Fallunterscheidung über die Tupelform des Knotens beschrieben.

\begin{listing}[htbp]
\caption{Arithmetische Auswertung in Python}
\label{lst:ply-evaluate}
\begin{minted}{python}
def evaluate(expr: NestedTuple, Values: dict[str, Number]) -> Number:
    match expr:
        case int():
            return expr
        case str():
            return Values[expr] 
        case ('call', 'read'):
            return int(input('Please enter a natural number: '))
        case ('call', 'print', expr):
            print(evaluate(expr, Values))
            return 0;
        case ('+', lhs, rhs):
            return evaluate(lhs, Values) + evaluate(rhs, Values)
        case ('-', lhs, rhs):
            return evaluate(lhs, Values) - evaluate(rhs, Values)
        case ('*', lhs, rhs):
            return evaluate(lhs, Values) * evaluate(rhs, Values)
        case ('/', lhs, rhs):
            return evaluate(lhs, Values) / evaluate(rhs, Values)
        case ('%', lhs, rhs):
            return evaluate(lhs, Values) % evaluate(rhs, Values)
        case _:
            assert False, f'{expr} unexpected'
\end{minted}
\end{listing}

Die TypeScript-Version bildet dieselbe rekursive Struktur nach, ergänzt jedoch explizite Prüfungen auf undefinierte Variablen sowie Fehlerbehandlung bei Division durch Null.

\begin{listing}[htbp]
\caption{Arithmetische Auswertung in TypeScript}
\label{lst:ts-evaluate}
\begin{minted}{typescript}
evaluate = (node: AST, env: Environment): number => {
    if (typeof node === "number") return node;
    if (typeof node === "string") {
        const val = env.get(node);
        if (val === undefined) throw new Error(`Undefined var ${node}`);
        return val;
    }

    if (Array.isArray(node)) {
        const tag = node[0];

        if (["+", "-", "*", "/", "%"].includes(tag)) {
             const leftNode = node[1];
             const rightNode = node[2];

             if (leftNode !== undefined && rightNode !== undefined) {
                 const l = evaluate(leftNode, env);
                 const r = evaluate(rightNode, env);
                 switch(tag) {
                     case "+": return l + r;
                     case "-": return l - r;
                     case "*": return l * r;
                     case "/": 
                         if (r === 0) throw new Error("Division by zero");
                         return Math.floor(l / r);
                     case "%": 
                         if (r === 0) throw new Error("Modulo by zero");
                         return l % r;
                 }
             }
        }

        if (tag === "Call") {
             const fn = node[1];
             const argsNode = node[2]; 

             if (fn === "print" && argsNode !== undefined) {
                 if (Array.isArray(argsNode) && argsNode[0] === '.') {
                     const actualArg = argsNode[1];
                     if (actualArg !== undefined) {
                        console.log(">> STDOUT:", evaluate(actualArg, env));
                     }
                 }
                 return 0;
             }
             if (fn === "read") {
                 const input = inputStream.shift();
                 const val = input !== undefined ? Number(input) : 0;
                 console.log("<< STDIN:", val);
                 return val;
             }
        }
    }
    throw new Error(`Invalid expression node: ${JSON.stringify(node)}`);
};
\end{minted}
\end{listing}

Die prinzipielle Auswertungslogik bleibt damit erhalten. Auffällig ist jedoch, dass die TypeScript-Version die Division mit \texttt{Math.floor(l / r)} als ganzzahlige Division realisiert, während in Python direkt der Operator \texttt{/} verwendet wird. Damit zeigt sich, dass eine Portierung nicht nur strukturelle, sondern auch semantische Detailentscheidungen umfasst.

\subsection{Auswertung boolescher Bedingungen}

Die boolesche Auswertung ist in beiden Implementierungen analog aufgebaut. Vergleichsoperatoren werden rekursiv ausgewertet und liefern einen Wahrheitswert zurück.

\begin{listing}[htbp]
\caption{Boolesche Auswertung in Python}
\label{lst:ply-evaluate-bool}
\begin{minted}{python}
def evaluate_bool(expr: NestedTuple, Values: dict[str, Number]) -> bool:
    match expr:
        case ('==', lhs, rhs):
            return evaluate(lhs, Values) == evaluate(rhs, Values)
        case ('!=', lhs, rhs):
            return evaluate(lhs, Values) != evaluate(rhs, Values)
        case ('<=', lhs, rhs):
            return evaluate(lhs, Values) <= evaluate(rhs, Values)
        case ('>=', lhs, rhs):
            return evaluate(lhs, Values) >= evaluate(rhs, Values)
        case ('<', lhs, rhs):
            return evaluate(lhs, Values) <  evaluate(rhs, Values)
        case ('>', lhs, rhs):
            return evaluate(lhs, Values) >  evaluate(rhs, Values)
        case _:
            assert False, f'{expr} unexpected'
\end{minted}
\end{listing}

\begin{listing}[htbp]
\caption{Boolesche Auswertung in TypeScript}
\label{lst:ts-evaluate-bool}
\begin{minted}{typescript}
evaluateBool = (node: AST, env: Environment): boolean => {
    if (Array.isArray(node)) {
        const tag = node[0];
        if (["==", "!=", "<", ">", "<=", ">="].includes(tag)) {
             const leftNode = node[1];
             const rightNode = node[2];

             if (leftNode !== undefined && rightNode !== undefined) {
                 const l = evaluate(leftNode, env);
                 const r = evaluate(rightNode, env);
                 switch(tag) {
                     case "==": return l == r;
                     case "!=": return l != r;
                     case "<":  return l < r;
                     case ">":  return l > r;
                     case "<=": return l <= r;
                     case ">=": return l >= r;
                 }
             }
        }
    }
    throw new Error("Invalid boolean expression");
};
\end{minted}
\end{listing}

Hier zeigt sich besonders deutlich, dass die eigentliche Interpretersemantik weitgehend unverändert bleibt. Unterschiede ergeben sich vor allem aus der konkreten Form der AST-Repräsentation und aus den sprachspezifischen Mitteln der Fallunterscheidung.

\subsection{Umgebungsmodell und Programmausführung}

Der Programmlauf basiert in beiden Notebooks auf einer Variablenumgebung, die Bezeichner auf numerische Werte abbildet. In Python wird hierfür ein Wörterbuch verwendet, in TypeScript eine \texttt{Map<string, number>}.

\begin{listing}[htbp]
\caption{Programmausführung in Python}
\label{lst:ply-main}
\begin{minted}{python}
def main(file):
    with open(file, 'r') as handle:
        program = handle.read() 
    stmnt = yacc.parse(program)
    print(stmnt)
    Values = {}
    execute(stmnt, Values)
\end{minted}
\end{listing}

\begin{listing}[htbp]
\caption{Programmausführung in TypeScript}
\label{lst:ts-runprogram}
\begin{minted}{typescript}
type Environment = Map<string, number>;
let inputStream: string[] = [];

function runProgram(fileName: string, inputs: string[] = []) {

Gerade an dieser Stelle wird ein praktischer Unterschied sichtbar: Während die Python-Version interaktive Eingaben direkt über \texttt{input(...)} verarbeitet, simuliert die TypeScript-Version Eingaben über einen vorbereiteten \texttt{inputStream}. Dies erleichtert insbesondere in Notebook-Umgebungen reproduzierbare Testläufe.

\subsection{Ergebnisse der Case Study}

Die Gegenüberstellung der restlichen Notebook-Bestandteile bestätigt die zentrale Beobachtung dieser Fallstudie: Die TypeScript-Version übernimmt die grundlegende Interpreterarchitektur des Python-Originals, modelliert die einzelnen Verarbeitungsschritte jedoch expliziter und modularer. Besonders deutlich wird dies an der Trennung von CST und AST, an der typisierten Beschreibung der Baumstruktur sowie an der systematischeren Behandlung rekursiver Listen und Laufzeitfehler.

Gleichzeitig zeigt der Vergleich der konkreten Codeausschnitte, dass die Portierung nicht nur eine Übertragung syntaktischer Konstrukte darstellt. Vielmehr werden bei der Umsetzung in TypeScript Entscheidungen über Abstraktionsniveau, Fehlerbehandlung und teilweise auch über semantische Details getroffen. Die TypeScript-Variante erscheint dadurch als stärker strukturierte und typisierte Rekonstruktion derselben interpretertechnischen Grundidee.
