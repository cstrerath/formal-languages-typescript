\section{Theoretische Grundlagen}
\label{sec:grundlagen}

Dieses Kapitel legt das theoretische Fundament für die Transformationsstrategie. Ein besonderer Schwerpunkt liegt auf der Analyse der Typsysteme, da die Diskrepanz zwischen der dynamischen Natur von Python und der statischen Analyse von TypeScript die zentrale Motivation für die Portierung darstellt.

\subsection{Typentheorie und Datenmodellierung}
\label{subsec:type_theory}

Ein Typsystem ist ein formales logisches System, das Programmkonstrukten Eigenschaften (Typen) zuweist, um das Programmverhalten zu klassifizieren. Nach Pierce gilt ein Programm als typsicher, wenn wohlgetypte Programme nicht in einen undefinierten Zustand geraten (\enquote{well-typed programs do not go wrong}) \autocite[S.1ff]{pierce2002tapl}.

In der Typentheorie wird Typsicherheit häufig durch zwei Eigenschaften charakterisiert \autocite[S.95ff]{pierce2002tapl}:
\begin{itemize}
	\item \textbf{Progress:} Ein wohlgetypter Ausdruck ist entweder bereits ein Wert oder kann einen weiteren Auswertungsschritt durchführen.
	\item \textbf{Preservation:} Wenn ein wohlgetypter Ausdruck einen Auswertungsschritt macht, behält das Resultat diesen Typ bei.
\end{itemize}

\subsubsection{Dynamische Typisierung in Python (Duck Typing)}
Python folgt dem Paradigma der dynamischen Typisierung. Typen sind hierbei Eigenschaften von Werten (Laufzeitobjekten), nicht von Variablen. Eine Variable ist lediglich ein Name, der an ein beliebiges Objekt gebunden werden kann. Die Typprüfung erfolgt implizit zur Laufzeit: Eine Operation (z.\,B. \texttt{obj.method()}) ist zulässig, wenn das Objekt zum Zeitpunkt des Aufrufs über das entsprechende Attribut verfügt.

Dieses Konzept wird als \textbf{Duck Typing} bezeichnet (\enquote{If it walks like a duck and quacks like a duck, it is a duck}).
Während dies maximale Flexibilität und schnelle Prototypenentwicklung ermöglicht, birgt es im Kontext compiler-naher Softwarearchitekturen signifikante Risiken:
\begin{enumerate}
	\item \textbf{Späte Fehlererkennung:} Inkonsistenzen in Datenstrukturen (z.\,B. ein fehlendes Feld in einem AST-Knoten) führen erst dann zu einem \texttt{AttributeError}, wenn der entsprechende Codepfad zur Laufzeit durchlaufen wird.
	\item \textbf{Implizite Annahmen:} Die Struktur komplexer Objekte (z.\,B. \enquote{Ein Token ist ein Tupel aus Typ und Wert}) muss durch Dokumentation oder Konvention gesichert werden, da die Sprache sie nicht erzwingt.
\end{enumerate}

Zwar bietet Python mit \href{http://mypy-lang.org/}{\texttt{mypy}} ein Werkzeug für optionale statische Typannotationen (PEP 484), dessen Analysefähigkeit ist jedoch begrenzt. Insbesondere bei der Flussanalyse (Control Flow) kann \texttt{mypy} oft nicht automatisch deduzieren, dass ein Typ nach einer Prüfung sicher ist, was zu erzwungenen \texttt{cast}-Operationen oder \texttt{\# type: ignore}-Kommentaren führt.

\subsubsection{Statische Typisierung und Control Flow Analysis in TypeScript}
Im Gegensatz dazu führt TypeScript eine strikte statische Typprüfung vor der Ausführung durch. Ein zentrales Unterscheidungsmerkmal zu traditionellen statischen Sprachen (wie Java) ist dabei die Fähigkeit zur **Control Flow Analysis (CFA)**. Der Compiler analysiert den logischen Fluss des Codes, um den Typ einer Variable an jeder Stelle im Code so präzise wie möglich zu bestimmen.

Dieses Verfahren wird als **Type Narrowing** bezeichnet. Aus einem breiten Typ (z.\,B. \texttt{string | number}) wird durch Bedingungen ein engerer Typ gefiltert \autocite{typescript_handbook_narrowing}.

\subsubsection{Type Narrowing und Discriminated Unions}
Eine zentrale Stärke von TypeScript ist die Fähigkeit, Typen innerhalb von Kontrollstrukturen zu verfeinern. Dieses als \emph{Narrowing} bezeichnete Verfahren ermöglicht es dem Compiler, den Typ einer Variable in einem bestimmten Codeblock präzise einzugrenzen.

Besonders relevant ist dies bei der Verwendung von \emph{Union Types} in Kombination mit Klassen. Der Operator \texttt{instanceof} fungiert in TypeScript als sogenannter \emph{Type Guard}. Listing~\ref{lst:narrowing_generic} illustriert dieses Prinzip anhand einer rekursiven Ausdrucksstruktur.

\begin{listing}[H]
	\caption{Type Narrowing mittels \texttt{instanceof} und Control Flow Analysis}
	\label{lst:narrowing_generic}
	\begin{minted}{typescript}
class Value { 
	constructor(public val: number) {} 
}

class Add { 
	constructor(public left: Expr, public right: Expr) {} 
}

type Expr = Value | Add;

function eval(e: Expr): number {
	if (e instanceof Value) {
		return e.val;
	}
	
	return eval(e.left) + eval(e.right);
}
	\end{minted}
\end{listing}

Der Code demonstriert das Zusammenspiel mehrerer TypeScript-Konzepte. Die Klassen \texttt{Value} und \texttt{Add} definieren die Struktur der Daten, wobei \texttt{Value} einen primitiven Wert hält und \texttt{Add} rekursiv auf zwei weitere Ausdrücke verweist. Der Typ \texttt{Expr} wird als Vereinigung (\emph{Union Type}) dieser beiden Klassen definiert, sodass eine Variable dieses Typs zur Laufzeit entweder eine Instanz von \texttt{Value} oder von \texttt{Add} sein kann.

Innerhalb der Funktion \texttt{eval} dient die Prüfung mittels \texttt{instanceof} als Type Guard. Der Compiler erkennt, dass im ersten \texttt{if}-Block die Variable \texttt{e} zwingend vom Typ \texttt{Value} ist, wodurch der Zugriff auf die Eigenschaft \texttt{val} statisch erlaubt wird. Im weiteren Verlauf greift die \emph{Control Flow Analysis}: Da der erste Pfad die Funktion durch ein \texttt{return} verlässt, schlussfolgert der Compiler für den verbleibenden Code, dass \texttt{e} nur noch vom Typ \texttt{Add} sein kann. Ein expliziter \texttt{else}-Block oder eine erneute Prüfung sind somit überflüssig; der Zugriff auf \texttt{left} und \texttt{right} ist typsicher gewährleistet.

Dies unterscheidet sich fundamental von Python, wo ein fehlendes Attribut erst zur Laufzeit einen Fehler verursachen würde. In TypeScript garantiert das Typsystem die strukturelle Integrität der Fallunterscheidung bereits vor der Ausführung.

\subsubsection{Strukturelle Typisierung (Structural Typing)}
Ergänzend zum Narrowing ist die \emph{strukturelle Typisierung} von TypeScript hervorzuheben. Im Gegensatz zu nominalen Systemen (wie Java oder C\#) entscheidet nicht der Name einer Klasse oder eine explizite \texttt{implements}-Klausel über die Kompatibilität, sondern ausschließlich die Form (\emph{Shape}) des Objekts.

Sei $A$ der Typ des zuzuweisenden Objekts (Kandidat) und $B$ der erwartete Zieltyp (Interface oder Klasse). Weiterhin sei $p$ eine beliebige Eigenschaft (Property) oder Methode eines Typs.
Der Typ $A$ gilt formal als kompatibel zu (oder Subtyp von) $B$, notiert als $A \subseteq B$, wenn $A$ alle Eigenschaften von $B$ besitzt und deren Typen kovariant sind:

\begin{equation}
	A \subseteq B \iff \forall p \in \text{properties}(B): p \in \text{properties}(A) \land \text{type}_A(p) \subseteq \text{type}_B(p)
	\label{eq:structural_typing}
\end{equation}

In Worten: Damit $A$ anstelle von $B$ verwendet werden kann, muss $A$ mindestens alle Eigenschaften besitzen, die in $B$ definiert sind. Zusätzliche Eigenschaften in $A$ stören die Kompatibilität nicht.

Listing~\ref{lst:structural_typing} verdeutlicht dieses Prinzip. Ein Interface definiert lediglich eine Anforderung an die Struktur. Jedes Objekt, das diese Anforderung erfüllt, wird vom Typsystem akzeptiert.

\begin{listing}[H]
	\caption{Implizite Kompatibilität ohne explizite Implementierung}
	\label{lst:structural_typing}
	\begin{minted}{typescript}
interface Vector2D {
	x: number;
	y: number;
}

function calculateLength(v: Vector2D): number {
	return Math.sqrt(v.x * v.x + v.y * v.y);
}

class Particle {
	constructor(
	public x: number, 
	public y: number, 
	public velocity: number
	) {}
}

const p = new Particle(3, 4, 10);
const len = calculateLength(p);
const len2 = calculateLength({ x: 5, y: 12 });
	\end{minted}
\end{listing}

Im gezeigten Beispiel wird die Klasse \texttt{Particle} ($A$) als Eingabe von der Funktion \texttt{calculateLength} akzeptiert, die \texttt{Vector2D} ($B$) erwartet. Da eine Instanz von \texttt{Particle} die geforderten Eigenschaften \texttt{x} und \texttt{y} besitzt (und mit \texttt{velocity} sogar zusätzliche Eigenschaften), gilt die Bedingung $A \subseteq B$ als erfüllt.

Dies erleichtert die Portierung von Python-Code erheblich, da Pythons \emph{Duck Typing}-Philosophie (\enquote{Wenn es wie eine Ente läuft, ist es eine Ente}) direkt auf TypeScripts Interfaces abgebildet werden kann. Ein Algorithmus, der beispielsweise erwartet, dass ein Objekt über eine \texttt{hashCode}-Methode verfügt, kann in TypeScript durch ein entsprechendes Interface typisiert werden, ohne dass die verwendeten Klassen in eine starre Vererbungshierarchie gezwungen werden müssen.

Obwohl die explizite Verwendung von \texttt{implements} für die Typkompatibilität technisch nicht erforderlich ist, wird sie in der Praxis dennoch empfohlen. Sie verlagert die Fehlererkennung von der Aufrufstelle (\emph{Call Site}) zur Definitionsstelle (\emph{Definition Site}). Würde in der Klasse \texttt{Particle} beispielsweise die Eigenschaft \texttt{x} versehentlich fehlen, würde der Compiler dies ohne \texttt{implements} erst beim Aufruf von \texttt{calculateLength(p)} bemängeln. Mit einer expliziten Klausel (\texttt{class Particle implements Vector2D}) wird der Fehler direkt an der Klasse markiert, was die Intention des Entwicklers dokumentiert und die Wartbarkeit erhöht.

\subsection{Realisierung von Wertsemantik in Laufzeitumgebungen}
\label{subsec:value_semantics_impl}

Ein zentrales Problem bei der Portierung von Python nach TypeScript ist die Diskrepanz im Identitätsbegriff. Während Python für Tupel und unveränderliche Mengen (\texttt{frozenset}) automatisch \emph{Wertsemantik} (Value Semantics) anwendet, basieren JavaScript-Container (\texttt{Set}, \texttt{Map}) bei Objekten strikt auf \emph{Referenzsemantik}. Zwei strukturell identische Zustände eines Automaten gelten in der JavaScript-Laufzeitumgebung als ungleich, sofern sie nicht physikalisch denselben Speicherbereich belegen. Diese Problematik wurde in Abbildung~\ref{fig:semantics_comparison} im Abschnitt~\ref{subsec:problemstellung} visualisiert.

Um die Algorithmen der theoretischen Informatik korrekt abzubilden, muss daher eine Datenstruktur implementiert werden, die Objekte anhand ihres Inhalts identifiziert. Hierfür existieren in der Informatik verschiedene algorithmische Standardverfahren.

\subsubsection{Vergleich der Implementierungsstrategien}
Zur Auswahl einer geeigneten Datenstruktur werden im Folgenden drei klassische Ansätze verglichen: Sortierte Arrays, Balancierte Suchbäume sowie Hash-Tabellen.

Ein zentrales Unterscheidungsmerkmal ist die benötigte Schnittstelle der Schlüsselobjekte. Vergleichsbasierte Verfahren setzen voraus, dass Schlüssel eine totale Ordnung besitzen (Interface \texttt{Comparable} / \texttt{compareTo}), wohingegen Hash-Verfahren lediglich die Prüfung auf Äquivalenz und die Berechnung eines Streuwertes erfordern (Interface \texttt{Equality} / \texttt{hashCode}).

\subsubsection{Bewertung der Verfahren}

Die Analyse der asymptotischen Laufzeiten zeigt deutliche Unterschiede in der Eignung für generische Bibliotheken.

\emph{Geordnete Arrays} ermöglichen durch die Binäre Suche zwar effiziente Lesezugriffe ($O(\log N)$) und sind extrem speichereffizient, da sie ohne Zeiger auskommen. Sie disqualifizieren sich jedoch für schreibintensive Szenarien. Sobald Mengen dynamisch wachsen, erfordert das Einfügen an einer beliebigen Stelle das Verschieben aller nachfolgenden Elemente im Speicher. Dies führt zu linearen Kosten ($O(N)$).
In anwendungsnahen Szenarien, wie etwa einem Backtracking-Algorithmus zum Lösen von Sudokus, werden millionenfach Zwischenzustände in Mengen eingefügt und wieder entfernt. Ein Aufwand von $O(N)$ pro Schritt würde die Gesamtlaufzeit hier exponentiell in die Höhe treiben.

\emph{Balancierte Suchbäume} garantieren zwar stabile logarithmische Laufzeiten für alle Operationen, weisen jedoch Nachteile auf technischer Ebene auf. Zum einen ist die Speicherstruktur ungünstig für moderne Hardware: Da Knoten einzeln im Heap allokiert und über Zeiger verknüpft sind, liegen logisch benachbarte Daten oft physikalisch weit verstreut. Dies führt beim Traversieren zu häufigen Cache-Misses \autocite[S. 301]{cormen2022introduction}.
Zum anderen stellt die Anforderung einer totalen Ordnung (\texttt{compareTo}) eine signifikante Hürde dar. Für komplexe, verschachtelte Objekte ist es oft schwierig oder rechenintensiv, eine konsistente Totalordnung zu definieren.

\emph{Hash-Tabellen} bieten mit einer amortisierten Laufzeit von $O(1)$ die theoretisch beste Performance für große, dynamische Mengen. Das Risiko von Kollisionen existiert, lässt sich durch qualitativ hochwertige Hash-Funktionen aber mitigieren. Entscheidend ist die Einfachheit: Durch den Verzicht auf komplexe Rebalancierungen und die mögliche Nutzung linearer Speicherbereiche profitieren Hash-Tabellen maximal von CPU-Caches \autocite[S. 301]{cormen2022introduction}.

\subsubsection{Anforderungen an Hash-Funktionen}
Eine Hash-Funktion $h: D \rightarrow I$ bildet Elemente aus einer theoretisch unendlichen Domäne $D$ (z.\,B. alle möglichen Zeichenketten oder Objektgraphen) auf einen endlichen Indexbereich $I = \{0, \dots, m-1\}$ ab. Damit eine Hash-Tabelle effizient arbeitet, muss die Funktion folgende theoretische Anforderungen erfüllen:

\begin{itemize}
	\item \textbf{Determinismus:} Für zwei identische Objekte $x$ und $y$ muss zwingend gelten: $x = y \implies h(x) = h(y)$. Dies ist trivial für primitive Datentypen, stellt jedoch bei rekursiven Strukturen eine Herausforderung dar. Da Standard-Sets keine definierte Iterationsreihenfolge besitzen, muss die Hash-Funktion kommutativ sein (d.\,h. unabhängig von der Reihenfolge der Elemente), oder die Elemente müssen vor dem Hashing in eine kanonische Ordnung gebracht werden.
	\item \textbf{Uniformität:} Die Wahrscheinlichkeit, dass ein beliebiger Schlüssel auf einen bestimmten Index abgebildet wird, sollte für alle Indizes gleich sein ($P(h(k) = i) \approx 1/m$). Eine ungleichmäßige Verteilung führt zu Häufungen (Clustering), was die Laufzeit der Hash-Tabelle im schlimmsten Fall auf $O(N)$ degradieren lässt \autocite[S. 283]{cormen2022introduction}.
	\item \textbf{Effizienz:} Die Berechnung von $h(x)$ muss schnell erfolgen. In der Praxis kommen oft Verfahren wie \emph{FNV-1a} oder \emph{MurmurHash} zum Einsatz, die durch Bit-Operationen (XOR, Shift) und Multiplikationen mit Primzahlen eine gute Durchmischung bei geringer CPU-Last erreichen.
\end{itemize}

\subsubsection{Strategien der Kollisionsauflösung}
Da die Domäne $D$ meist wesentlich größer ist als der Indexbereich $I$, sind Kollisionen ($h(x) = h(y)$ mit $x \neq y$) nach dem Schubfachprinzip unvermeidbar. Zur Auflösung dieser Konflikte unterscheidet die Literatur zwei Hauptklassen von Verfahren:

\paragraph{Separate Chaining (Verkettung)}
Beim Chaining enthält jeder Slot der Hash-Tabelle einen Zeiger auf eine dynamische Datenstruktur (meist eine einfach verkettete Liste), die alle Elemente aufnimmt, die auf diesen Index hashen (siehe Abbildung~\ref{fig:chaining}).
Der Vorteil dieses Verfahrens ist seine Robustheit: Der Füllgrad der Tabelle kann theoretisch über 100\,\% steigen, ohne dass die Tabelle vollständig blockiert. Der Nachteil liegt im Speicheroverhead durch die Zeiger und der schlechten Cache-Lokalität, da die Listen-Knoten verstreut im Heap liegen können.

\paragraph{Open Addressing (Offene Adressierung)}

\subsubsection{Erhalt der Invarianten beim Löschen (Backshifting)}
Eine besondere Herausforderung bei der offenen Adressierung stellt das Entfernen von Elementen dar. Wird ein Element naiv gelöscht, entsteht eine Lücke (\texttt{null}) im Cluster. Nachfolgende Elemente, die aufgrund einer Kollision hinter dieser Lücke platziert wurden, wären bei einer erneuten Suche nicht mehr auffindbar, da der Suchalgorithmus bei der ersten Lücke abbricht.

Klassische Implementierungen verwenden daher oft \enquote{Tombstones} (spezielle Marker für gelöschte Felder). Diese führen jedoch langfristig zu einer Verschmutzung der Tabelle und verlangsamen Fehlsuchen, da der Algorithmus über Tombstones hinwegspringen muss.

Um die Performance auch bei intensiven Modifikationen stabil zu halten, kommt in dieser Arbeit das Verfahren des \textbf{Backshift Deletion} (auch bekannt als \emph{Robin Hood Deletion}) zum Einsatz. Listing~\ref{lst:backshift_algo} zeigt eine mögliche Implementierung.

\begin{listing}[H]
	\caption{Algorithmus zur Rückverschiebung (Backshift Deletion), adaptiert nach \autocite[S. 302]{cormen2022introduction}}
	\label{lst:backshift_algo}
	\begin{minted}{typescript}
function deleteAtIndex(index: number): void {
	table[index] = null;
	
	let hole = index;
	let scanner = (hole + 1) % capacity;
	
	while (table[scanner] !== null) {
		const element = table[scanner];
		const ideal = hash(element.key);
		
		const distToHole = (hole - ideal + capacity) % capacity;
		const distToScanner = (scanner - ideal + capacity) % capacity;
		
		if (distToHole < distToScanner) {
			table[hole] = element;
			table[scanner] = null;
			hole = scanner;
		}
		
		scanner = (scanner + 1) % capacity;
	}
}
	\end{minted}
\end{listing}

Der Algorithmus stellt die Invariante des linearen Sondierens wieder her, indem er Lücken schließt. Die Logik lässt sich wie folgt analysieren:

\begin{itemize}
	\item \textbf{Zeile 2:} Das Element an der übergebenen Position wird physisch entfernt. Es entsteht eine Lücke (Variable \texttt{hole}).
	\item \textbf{Zeile 5:} Ein zweiter Zeiger (\texttt{scanner}) beginnt unmittelbar nach der Lücke und durchläuft zyklisch den restlichen Cluster.
	\item \textbf{Zeile 11--12:} Hier erfolgt die Kernprüfung. Es werden zwei Distanzen berechnet: Wie weit ist die Lücke vom idealen Hash-Index des Elements entfernt (\texttt{distToHole}) und wie weit ist die aktuelle Position entfernt (\texttt{distToScanner}). Die Modulo-Arithmetik behandelt dabei den zyklischen Überlauf am Array-Ende.
	\item \textbf{Zeile 14:} Ist die Lücke näher an der idealen Position als der aktuelle Platz ($distToHole < distToScanner$), bedeutet dies, dass das Element ursprünglich an die Stelle der Lücke (oder davor) wollte, aber verdrängt wurde.
	\item \textbf{Zeile 15--17:} Das Element wird in die Lücke zurückgeschoben. Dadurch wandert die Lücke nun an die Position des Scanners. Der Prozess wiederholt sich rekursiv für die neue Lückenposition, bis das Ende des Clusters erreicht ist.
\end{itemize}

Die Terminierung des Algorithmus ist durch die Eigenschaften des Linear Probing garantiert, sofern der Füllgrad der Tabelle unter $100\,\%$ gehalten wird (typischerweise $\alpha \le 0.75$).
Ein \emph{Cluster} ist in einer Hash-Tabelle definiert als eine zusammenhängende Sequenz von belegten Slots, die von leeren Slots (\texttt{null}) begrenzt wird. Da der \texttt{scanner}-Zeiger den Cluster ab der Lücke zyklisch durchläuft und sich nur vorwärts bewegt, muss er zwangsläufig das Ende des Clusters erreichen. Da die Tabelle nicht voll sein kann, existiert mindestens ein \texttt{null}-Feld, welches die \texttt{while}-Schleife in Zeile 7 abbricht. Ein unendlicher Loop ist somit ausgeschlossen.

Das Verfahren hält die Tabelle kompakt und eliminiert die Notwendigkeit von Tombstones vollständig. Zwar sind die Kosten einer einzelnen Löschoperation im Worst-Case höher als beim bloßen Markieren, doch wird dies durch die verbesserte Cache-Lokalität und schnellere Suchzeiten (Misses) bei moderaten Füllgraden kompensiert.


\subsection{Laufzeitumgebungen und WebAssembly}
\label{subsec:wasm_theory}

Ein wesentliches Ziel bei der Bereitstellung der Projektresultate ist die barrierefreie Nutzung der Visualisierungskomponenten in interaktiven Umgebungen wie \emph{Jupyter Notebooks}. Traditionell erfordert die Darstellung von Automaten-Graphen via \emph{Graphviz} die manuelle Installation nativer Binärdateien auf dem Host-System. Insbesondere unter Windows-Umgebungen führt die notwendige Konfiguration von Systempfaden häufig zu Instabilitäten und erschwert die Reproduzierbarkeit der Forschungsergebnisse.

Um diesen \enquote{Tricky Paths}-Problematiken zu begegnen, nutzt die Visualisierungslogik \ac{wasm}. Dabei handelt es sich um ein binäres Instruktionsformat für eine stapelbasierte virtuelle Maschine, das als portables Zielformat für die Kompilierung von Hochsprachen wie C++ dient \autocite{haas2017bringing}. 

In dieser Arbeit wird die Bibliothek \texttt{@viz-js/viz} eingesetzt, welche den ursprünglichen C-Code von Graphviz mittels \emph{Emscripten} zu \ac{wasm} kompiliert hat. Dies ermöglicht es, die komplexen Layout-Algorithmen (wie \emph{dot}) direkt als reguläre Abhängigkeit via \texttt{npm} zu beziehen und innerhalb der Node.js-Laufzeitumgebung des Jupyter-Kernels (\emph{tslab}) auszuführen. Damit wird die geforderte \Gls{interoperabilität} auf Ebene der Entwicklungsumgebung erreicht: Das System ist nach einem Standard-Setup (\texttt{conda} und \texttt{npm install}) sofort einsatzbereit, ohne dass externe Grafik-Tools auf Betriebssystemebene vorinstalliert sein müssen.
