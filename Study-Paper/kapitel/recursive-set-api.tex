\section{RecursiveSet: API und fachliche Modellierung}
\label{sec:recursive-set-api}

Wie im Kapitel~\ref{sec:einleitung} (siehe Listing~\ref{lst:set_python} und Listing~\ref{lst:set_ts_untyped}) ersichtlich, gibt es zwischen der Mengenverarbeitung in Python und TypeScript fundamentale Unterschiede. Dieses Kapitel erläutert zunächst die theoretischen Ursachen für dieses abweichende Verhalten. Darauf aufbauend wird gezeigt, wie diese Diskrepanzen mithilfe der speziell entwickelten Bibliothek \mintinline{text}{RecursiveSet} mitigiert werden können, um die aus Python vertraute Arbeitsweise wiederherzustellen.

\subsection{Mathematische Mengen und der Begriff der Wohlunterscheidbarkeit}
\label{subsec:rs-api-mathematische-mengen}

Was ist überhaupt eine Menge im mathematischen Sinne? Konsultiert man die gängige Fachliteratur, stößt man unweigerlich auf die klassische Definition von Georg Cantor: \enquote{Eine Menge ist eine Zusammenfassung bestimmter wohlunterschiedener Objekte unserer Anschauung oder unseres Denkens [...] zu einem Ganzen.} Die zentrale Frage für die Übertragung von Algorithmen in eine Programmiersprache lautet dabei: Wie genau ist \enquote{wohlunterscheidbar} definiert?

Genau in der Interpretation dieses Begriffs unterscheiden sich Python und TypeScript (bzw. die zugrunde liegende Laufzeitumgebung JavaScript) diametral. Es stehen sich zwei grundverschiedene Paradigmen gegenüber: die Unterscheidung nach dem inhaltlichen Wert (Wertsemantik) und die Unterscheidung nach der physischen Speicheradresse (Referenzsemantik).

\paragraph{Semantik-Bruch: Inhalt vs. Speicheradresse}
\label{par:rs-api-semantik-bruch}

In Python werden zusammengesetzte Datenstrukturen wie Tupel nach ihrem tatsächlichen Inhalt unterschieden. Fügen wir die Tupel \mintinline{python}{(1, 2)} und \mintinline{python}{(2, 1)} in eine Menge ein, erhalten wir mathematisch korrekt eine Menge mit zwei Elementen. Fragen wir die Menge anschließend, ob das Tupel \mintinline{python}{(2, 1)} enthalten ist, so lautet die Antwort \mintinline{python}{True}. Python erkennt, dass die Struktur und die inneren Werte identisch sind.

JavaScript hingegen geht deutlich maschinennäher vor. Erschaffen wir ein Array \mintinline{typescript}{[1, 2]}, so wird dieses zur Laufzeit an einer bestimmten Speicheradresse (beispielsweise \texttt{0xABCD}) abgelegt. Fügen wir dieses Array in ein natives \mintinline{typescript}{Set} ein, speichert JavaScript unter der Haube faktisch nur diese konkrete Speicheradresse ab. 

Der folgende direkte Code-Vergleich macht diesen Unterschied und seine Folgen deutlich:

\begin{listing}[H]
	\caption{Wohlunterscheidbarkeit bei Tupeln/Arrays: Python vs. TypeScript}
	\label{lst:set_wohlunterscheidbar_comparison}
	\begin{minted}{python}
# Python (mit mypy): Wertsemantik (Structural Equality)
# Mengen unterscheiden Elemente nach ihrem Inhalt.
zahlen_menge: set[tuple[int, int]] = set()

zahlen_menge.add((1, 2))
zahlen_menge.add((2, 1))

print(zahlen_menge)          # {(1, 2), (2, 1)}

# Eine neu erzeugte Instanz mit gleichem Inhalt wird als identisch erkannt:
print((2, 1) in zahlen_menge) # True
	\end{minted}
	
	\vspace{0.5em}
	
	\begin{minted}{typescript}
// TypeScript: Referenzsemantik (Reference Equality)
// Mengen unterscheiden Objekte nach ihrer Speicheradresse.
type ZahlenTupel = [number, number];
const zahlenMenge = new Set<ZahlenTupel>();

zahlenMenge.add([1, 2]); // Speichert z.B. Adresse 0xABCD
zahlenMenge.add([2, 1]); // Speichert z.B. Adresse 0xABCE

// Wir erzeugen ein NEUES Array mit exakt demselben Inhalt [2, 1].
// Dieses liegt an einer neuen Speicheradresse (z.B. 0xABCF).
// Da 0xABCF nicht in der Menge existiert, schlägt die Abfrage fehl:
console.log(zahlenMenge.has([2, 1])); // false
	\end{minted}
\end{listing}

Aus rein technischer Sicht ist die JavaScript-Herangehensweise durchaus valide und extrem schnell: Speicheradressen sind zweifellos \enquote{wohlunterscheidbar}. Für die praktische Anwendung in der theoretischen Informatik ist dieses Verhalten jedoch katastrophal. Algorithmen, die beispielsweise Zustandsübergänge (Tupel aus Zustand und Symbol) oder kartesische Produkte berechnen, generieren im Speicher fortlaufend neue Instanzen. Wenn diese von der Mengenstruktur nicht als inhaltliche Duplikate erkannt werden, schlagen Mitgliedschaftstests fehl und fundamentale Invarianten der Algorithmen brechen zusammen.

\paragraph{Mitigation durch \texttt{RecursiveSet}}
\label{par:rs-api-mitigation}

Um die Lehrnotebooks funktionsfähig nach TypeScript portieren zu können, muss die Laufzeitumgebung wieder dazu gezwungen werden, Mengen nach ihrem mathematischen Inhalt zu bewerten. Genau hier setzt die Bibliothek \mintinline{text}{RecursiveSet} an. 

Indem wir native Arrays in eine spezielle \mintinline{typescript}{Tuple}-Klasse kapseln, welche die nötigen Methoden bereitstellt, um den Inhalt sicher zu vergleichen (siehe Kapitel~\ref{sec:recursive-set-intern} zu den Interna), und \mintinline{typescript}{RecursiveSet} anstelle des nativen \mintinline{typescript}{Set} verwenden, verhält sich TypeScript wieder exakt wie Python:

\begin{listing}[H]
	\caption{Wiederherstellung der Wertsemantik durch \texttt{RecursiveSet}}
	\label{lst:rs_wertsemantik_loesung}
	\begin{minted}{typescript}
// TypeScript mit RecursiveSet: Der Inhalt ist wieder entscheidend
const zahlenMenge = new RecursiveSet<Tuple<[number, number]>>();

zahlenMenge.add(new Tuple(1, 2));
zahlenMenge.add(new Tuple(2, 1));

// Eine neue Instanz wird nun sicher anhand ihres Inhalts gefunden:
console.log(zahlenMenge.has(new Tuple(2, 1))); // true
	\end{minted}
\end{listing}

Mit diesem wiederhergestellten Fundament, der inhaltlichen Wohlunterscheidbarkeit, lassen sich nun die aus den Python-Notebooks bekannten, komplexeren mathematischen Mengenoperationen direkt auf TypeScript übertragen.
