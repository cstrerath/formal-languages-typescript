\section{Interne Architektur von RecursiveSet}
\label{sec:recursive-set-intern}

Während Kapitel~\ref{sec:recursive-set-api} die fachliche Programmierschnittstelle und die konzeptionelle Notwendigkeit einer wertsemantischen Mengenstruktur behandelte, fokussiert dieses Kapitel die zugrunde liegende technische Architektur. Dabei wird beleuchtet, wie die Bibliothek die Brücke zwischen der formalen mathematischen Theorie und der maschinennahen Ausführung in der JavaScript-Laufzeitumgebung (\gls{v8}) schlägt.

% !TEX root = ../master.tex

\subsection{Architektonische Grundentscheidungen}
\label{subsec:rs-intern-grundentscheidungen}

Die naheliegendste Frage zu Beginn einer solchen Portierung lautet: Warum wird das native \mintinline{typescript}{Set} von TypeScript nicht einfach in eine Wrapper-Klasse gewickelt, um das in Abschnitt~\ref{subsec:rs-api-wohlunterscheidbarkeit} beschriebene Problem der mangelnden Wertsemantik zu umgehen? Man könnte komplexe Objekte beispielsweise in \glslink{json}{JSON}-Strings umwandeln (\mintinline{typescript}{JSON.stringify}) und diese Strings als Repräsentation im nativen Set speichern. 

Ein solcher Ansatz scheitert jedoch an zwei fundamentalen Hürden: einer semantischen und einer architektonischen.

Die \textbf{semantische Hürde} betrifft das Wesen mathematischer Mengen. Mengen sind fachlich ungeordnet: Die Menge $\{1, 2\}$ ist mathematisch absolut identisch zur Menge $\{2, 1\}$. Eine Serialisierung mittels \mintinline{typescript}{JSON.stringify} ist jedoch strikt sequenziell. Sie würde die Repräsentationen \mintinline{text}{"[1,2]"} und \mintinline{text}{"[2,1]"} als zwei völlig unterschiedliche Zeichenketten und damit als zwei verschiedene Elemente werten. Um dies zu umgehen, müsste jede geschachtelte Menge vor jedem Lese- oder Schreibzugriff tiefenrekursiv sortiert werden.

Dies führt unweigerlich zur \textbf{architektonischen Hürde}: Zwar ist die native \mintinline{typescript}{JSON.stringify}-Funktion der JavaScript-Laufzeitumgebung hochgradig optimiert, doch das erzwungene vorherige Sortieren und das ständige Erzeugen neuer String-Objekte bei jedem noch so kleinen Zugriffsobjekt erfordert enorm viel \glslink{cpu}{CPU}-Zeit. Zudem belastet die ständige Erzeugung temporärer Strings die Speicherverwaltung massiv:

\begin{definition}
\label{def:garbage-collection}
\textbf{\glslink{garbagecollection}{Garbage Collection} (\gls{gc}):} Ein Prozess der Laufzeitumgebung (wie der \glslink{v8}{V8-Engine}), der den Arbeitsspeicher nach Objekten durchsucht, die vom Programm nicht mehr referenziert werden, um deren Speicherplatz wieder freizugeben. Werden in kurzen Abständen extrem viele temporäre Objekte (wie \glslink{json}{JSON}-Strings für Suchabfragen) erzeugt, muss die \gls{gc} den Programmablauf für Aufräumarbeiten pausieren (\enquote{Stop-the-World}-Pausen), was zu Latenzen und Einbrüchen in der Performance führt.
\end{definition}

Um sowohl die mathematische Gleichheit (ordnungsunabhängige Wertsemantik) als auch eine Zugriffszeit von im Durchschnitt \glslink{amortisiert}{amortisiert} $\mathcal{O}(1)$ (unter der Annahme einer guten Hash-Verteilung) zu garantieren, wurde \mintinline{typescript}{RecursiveSet} stattdessen als vollständige Eigenimplementierung entwickelt. Nur so erhält die Bibliothek die volle Kontrolle über den Hashing-Algorithmus (der bei Mengen ordnungsunabhängig arbeiten muss) und das Speicherlayout, um unnötige Objekt-Allokationen zu vermeiden.

\subsection{Die Hash-Engine (Zentrale Komponente)}
\label{subsec:rs-intern-hash-engine}

Ein zentraler Bestandteil jeder Hashtabelle ist die Hash-Engine. Ihre Aufgabe ist es, aus beliebigen, komplexen Werten deterministische Zahlen (Hash-Werte) zu generieren. Dieser Hash-Wert fungiert als mathematischer Wegweiser, der den Suchraum in der Tabelle massiv eingrenzt, bevor die aufwendigere inhaltliche Gleichheitsprüfung stattfindet.

Die Bibliothek nutzt die zentrale Funktion \mintinline{typescript}{getHashCode} als \textit{Dispatcher}. Ein Dispatcher (engl. für Verteiler oder Leitstelle) ist ein Architekturmuster, bei dem eine zentrale Funktion den eingehenden Datentyp untersucht und die Aufgabe an den jeweils spezialisierten, effizientesten Unteralgorithmus weiterleitet.

\subsubsection{Ganzzahlen und der ArrayBuffer-Trick für IEEE-754}
\label{subsubsec:rs-intern-hash-numbers}

Wie bereits in Definition~\ref{def:ts-number} (Kapitel \ref{sec:typisierung}) dargelegt, unterscheidet TypeScript auf Typ-Ebene nicht zwischen Ganzzahlen und Kommazahlen. Alle numerischen Werte werden nach dem \glslink{ieee}{IEEE}-754-Standard als 64-Bit-Fließkommazahlen behandelt. 

Für die Berechnung von Hash-Werten ist dies jedoch eine architektonische Hürde: Klassische bitweise Hash-Operationen (wie das exklusive ODER, \mintinline{typescript}{^}) arbeiten in JavaScript ausschließlich mit 32-Bit-Ganzzahlen. Wendet man sie direkt auf eine Fließkommazahl an, schneidet die Laufzeitumgebung die Nachkommastellen schlichtweg ab. Der Hash von \mintinline{typescript}{3.1} wäre demnach identisch mit dem Hash von \mintinline{typescript}{3.9}, was zu massiven Hash-Kollisionen führen würde.

Die Hash-Engine löst dieses Problem durch zwei strikt getrennte Ausführungspfade. Da dieser Algorithmus, siehe Listing~\ref{lst:rs-hashing-numbers}, sehr dicht und hardwarenah geschrieben ist, lohnt sich eine schrittweise Betrachtung.

\begin{listing}[ht]
    \caption{Hashing von numerischen Werten}
    \label{lst:rs-hashing-numbers}
    \begin{minted}{typescript}
const _buffer = new ArrayBuffer(8);
const _f64 = new Float64Array(_buffer);
const _i32 = new Int32Array(_buffer);

// Mixing constants (derived from MurmurHash3)
const HASH_C1 = 0xcc9e2d51;
const HASH_C2 = 0x1b873593;

function getHashCode(val: Value): number {
    if (typeof val === 'number') {
        // Fast Path: 32-Bit Ganzzahlen
        if ((val | 0) === val) {
            let h = val | 0;
            h = Math.imul(h ^ (h >>> 16), 0x85ebca6b);
            h = Math.imul(h ^ (h >>> 13), 0xc2b2ae35);
            return (h ^ (h >>> 16)) >>> 0;
        }

        // Slow Path: IEEE-754 Fließkommazahlen
        _f64[0] = val;
        const low = _i32[0];
        const high = _i32[1];

        return (Math.imul(low, HASH_C1) ^ Math.imul(high, HASH_C2)) >>> 0;
    }
    // ...
}
    \end{minted}
\end{listing}

\paragraph*{Schritt 1: Globale Speichervorbereitung (Zeile 1–7)}
Noch bevor die eigentliche Funktion aufgerufen wird, reserviert die Bibliothek mit \mintinline{typescript}{new ArrayBuffer(8)} einen 8 Bytes (64 Bits) großen, rohen Block im Arbeitsspeicher. Die typisierten Arrays \mintinline{typescript}{_f64} und \mintinline{typescript}{_i32} erzeugen dabei keine neuen Daten, sondern fungieren als zwei unterschiedliche \enquote{Linsen} (Sichten) auf exakt denselben Speicherblock. Wofür diese Linsen benötigt werden, zeigt sich im Slow Path. Zusätzlich werden die globalen Konstanten \mintinline{typescript}{HASH_C1} und \mintinline{typescript}{HASH_C2} definiert, die aus dem bewährten MurmurHash3-Algorithmus entlehnt sind \autocite{appleby2008murmurhash}.

\paragraph*{Schritt 2: Der Fast Path für Ganzzahlen (Zeile 11–17)}
Zunächst wird mittels der bitweisen Operation \mintinline{typescript}{(val | 0)} geprüft, ob die Zahl trotz des Fließkomma-Formats verlustfrei als 32-Bit-Ganzzahl behandelt werden kann. 

Ist dies der Fall, wird der Wert durch sogenanntes \textit{Avalanche-Mixing} (Lawineneffekt) vermischt. Warum ist das nötig? Würde man aufeinanderfolgende Zahlen (wie 1, 2 und 3) unverändert als Hash-Werte nutzen, entstünden in der Hashtabelle schnell sogenannte \textbf{Cluster} (Klumpen). Das bedeutet, dass viele Elemente direkt nebeneinander abgelegt werden. Beim späteren Einfügen von Elementen führte dies unweigerlich zu langen Suchen nach Plätzen, da der Algorithmus viele belegte Positionen überspringen muss, bevor er eine Lücke findet.

Um diese Klumpenbildung zu verhindern, nutzt der Code Konstanten wie \mintinline{typescript}{0x85ebca6b}. Das Präfix \mintinline{typescript}{0x} kennzeichnet dabei lediglich eine \textbf{hexadezimale} Zahl, eine kompakte Schreibweise für lange Bitmuster. Diese speziellen Bitmuster sorgen bei der Multiplikation für eine optimale Streuung: Ändert sich an der Eingabezahl nur ein einziges Bit, ändert sich durch den Lawineneffekt fast das gesamte Bitmuster des resultierenden Hash-Werts. So landen auch sehr ähnliche Zahlen auf völlig unterschiedlichen Tabellenplätzen.

\paragraph*{Schritt 3: Hardwarenahe Multiplikation mit \texttt{Math.imul}}
Auffällig ist in Zeile 14 und 15 die ständige Verwendung von \mintinline{typescript}{Math.imul}. Ein gewöhnliches Multiplikationszeichen (\mintinline{typescript}{*}) würde in JavaScript dazu führen, dass die erzeugten Bitmuster intern wieder als 64-Bit-Fließkommazahlen berechnet werden. Bei großen Multiplikatoren führt das unweigerlich zu Genauigkeitsverlusten (Rundungsfehlern) und macht die unteren Bits des Hash-Werts unbrauchbar. Die Funktion \mintinline{typescript}{Math.imul} (Integer Multiplication) zwingt die JavaScript-Engine stattdessen zu einer echten, hardwarenahen 32-Bit-Ganzzahl-Multiplikation. Bits, die durch einen Überlauf über die 32-Bit-Grenze hinausschießen, werden auf \glslink{cpu}{CPU}-Ebene einfach abgeschnitten, was für Hashing-Algorithmen das gewünschte Verhalten ist.

\paragraph*{Schritt 4: Der Slow Path für Fließkommazahlen (Zeile 19–24)}
Schlägt die Ganzzahl-Prüfung fehl, greift der Slow Path. Hier kommt nun ein architektonischer Ansatz zum Einsatz, der an ein \mintinline{c}{union}-Konstrukt aus der Sprache C erinnert. Um die gesamten 64 Bits der Fließkommazahl (Vorzeichen, Exponent und Mantisse) für das Hashing zu nutzen, müssen diese in zwei 32-Bit-Hälften zerlegt werden. 

Anstatt die Zahl durch teure mathematische Operationen zu zerspalten, greift die Bibliothek auf den in Schritt 1 vorbereiteten \mintinline{typescript}{ArrayBuffer} zurück:
\begin{enumerate}
    \item Die Fließkommazahl wird durch die 64-Bit-Linse (\mintinline{typescript}{_f64[0] = val}) in Zeile 20 nativ in den Speicher geschrieben.
    \item Anschließend werden in den Zeilen 21 und 22 exakt dieselben 64 Bits durch die Ganzzahl-Linse (\mintinline{typescript}{_i32}) ohne jegliche Konvertierung als zwei fertige 32-Bit-Blöcke (\mintinline{typescript}{low} und \mintinline{typescript}{high}) wieder ausgelesen.
\end{enumerate}
Dieses direkte Lesen und Schreiben von Speicherbits erzeugt keine temporären Objekte und entlastet den \glslink{garbagecollection}{Garbage Collector} somit vollständig. Zum Abschluss werden beide Hälften in Zeile 24 über \mintinline{typescript}{Math.imul} mit den globalen Konstanten \mintinline{typescript}{HASH_C1} und \mintinline{typescript}{HASH_C2} überkreuz multipliziert und per exklusivem ODER vereint. Dadurch ist sichergestellt, dass selbst winzigste Änderungen im Exponenten oder in der Mantisse der Fließkommazahl einen neuen Hash-Wert generieren.

\subsubsection{Hashing von Strings und strukturellen Objekten}
\label{subsubsec:rs-intern-hash-strings-objects}

Bei Zeichenketten und komplexen Objekten greifen andere Mechanismen, die ebenfalls auf Performance und Speicherschonung ausgelegt sind, wie in Listing~\ref{lst:rs-hashing-strings-objects} dargestellt.

\begin{listing}[ht]
    \caption{Hashing von Strings und strukturellen Objekten}
    \label{lst:rs-hashing-strings-objects}
    \begin{minted}{typescript}
    if (typeof val === 'string') {
        let h = 0x811c9dc5;
        const len = val.length;
        for (let i = 0; i < len; i++) {
            h ^= val.charCodeAt(i);
            h = Math.imul(h, 0x01000193);
        }
        return h >>> 0;
    }

    if (val && typeof val === 'object') {
        return val.hashCode;
    }
    \end{minted}
\end{listing}

Für \textbf{Strings} verwendet die Bibliothek eine Variante des \textit{FNV-1a}-Algorithmus (Fowler–Noll–Vo). Die Zeichenkette wird in einer Schleife durchlaufen und der numerische Wert jedes Zeichens (\mintinline{typescript}{charCodeAt}) wird über ein exklusives ODER (\mintinline{typescript}{^=}) in den Hash eingemischt. Der bewusste Verzicht auf Methoden, die den String in Arrays aufspalten oder neue Teilzeichenketten (Sub-Strings) erzeugen, stellt sicher, dass während der Schleife keine temporären Objekte im Arbeitsspeicher reserviert (alloziert) werden müssen. Diese sogenannte \textit{Allokationsfreiheit} entlastet den \glslink{garbagecollection}{Garbage Collector}.

Bei \textbf{komplexen Objekten} (wie verschachtelten Mengen oder Tupeln) delegiert die Hash-Engine die Verantwortung an das Objekt selbst, indem sie schlicht \mintinline{typescript}{val.hashCode} abruft. 

Damit der TypeScript-Compiler garantieren kann, dass diese Eigenschaft auf dem übergebenen Objekt überhaupt existiert, stützt sich die Bibliothek auf einen vertraglichen Rahmen: das interne \mintinline{typescript}{Structural}-Interface. Dieses ist in Listing~\ref{lst:rs-structural-interface-intern} dargestellt.

\begin{listing}[ht]
    \caption{Der Vertrag für komplexe Mengenelemente}
    \label{lst:rs-structural-interface-intern}
    \begin{minted}{typescript}
interface Structural {
    /**
     * Ein stabiler Hash-Wert für das Objekt. 
     * Der Zugriff auf diese Eigenschaft friert den Zustand des Objekts 
     * in rekursiven Strukturen in der Regel ein.
     */
    readonly hashCode: number;

    /**
     * Prüft die inhaltliche (strukturelle) Gleichheit mit einem anderen Objekt.
     */
    equals(other: unknown): boolean;

    /** * Liefert eine deterministische Textrepräsentation.
     */
    toString(): string;
}
    \end{minted}
\end{listing}

Wie in Abschnitt~\ref{subsubsec:interfaces-strukturell} im Rahmen des statischen Duck-Typings erläutert, fordert TypeScript keine starren Klassenhierarchien. Jedes Objekt, das die in Listing~\ref{lst:rs-structural-interface-intern} definierten Eigenschaften und Methoden bereitstellt, wird vom Typsystem automatisch als \mintinline{typescript}{Structural} anerkannt und kann in der Bibliothek als Schlüssel oder Mengenelement verwendet werden.

Dieser Vertrag definiert exakt die drei Säulen, die eine wertebasierte Hashtabelle benötigt:
\begin{enumerate}
    \item \textbf{\texttt{hashCode}:} Liefert den vorberechneten Hash-Wert zur schnellen Eingrenzung des Suchraums. Das Schlüsselwort \mintinline{typescript}{readonly} garantiert dem Typsystem, dass sich dieser Wert nach der Erstellung nicht mehr ändert. Der lesende Zugriff auf diese Eigenschaft ist zudem der Auslöser für den kaskadierenden Freeze-Mechanismus (siehe Abschnitt~\ref{subsec:rs-intern-freeze}).
    \item \textbf{\texttt{equals}:} Übernimmt die inhaltliche Gleichheitsprüfung, sobald der Hash-Wert eine mögliche Kollision oder einen Treffer anzeigt (siehe Abschnitt~\ref{subsec:rs-intern-areequal}).
    \item \textbf{\texttt{toString}:} Erzwingt eine konsistente Textrepräsentation, was für die deterministische Konsolenausgabe von tief verschachtelten Strukturen essenziell ist.
\end{enumerate}

Genau dieses Interface ist der Grund, warum sich die in Kapitel~\ref{sec:recursive-set-api} vorgestellten Klassen \mintinline{typescript}{Tuple} und \mintinline{typescript}{RecursiveMap} nahtlos, typsicher und ohne zusätzlichen Konvertierungsaufwand in die Hash-Architektur einfügen lassen.

\subsection{Speicherlayout (Structure of Arrays)}
\label{subsec:rs-intern-speicherlayout}

Ein klassischer, objektorientierter Entwurf für eine Hashtabelle würde Objekte der Form \mintinline{typescript}{{ value, hash }} anlegen. Diesen Ansatz nennt man \textit{\glslink{aos}{Array of Structures}} (\gls{aos}). Bei Tausenden von Zuständen in einem Automaten führt dies jedoch dazu, dass der Arbeitsspeicher fragmentiert. Die \gls{cpu} muss ständig an verschiedene Orte im \glslink{ram}{RAM} springen, um die Objekte zu lesen, was zusätzliche Taktzyklen erfordert.

Um die Cache-Lokalität der \glslink{cpu}{CPU} besser auszunutzen, wird von \mintinline{typescript}{RecursiveSet} ein Entwurfsmuster aus dem High-Performance-Computing implementiert:

\begin{definition}
\label{def:structure-of-arrays}
\textbf{\glslink{soa}{Structure of Arrays} (\gls{soa_acr}):} Bei diesem Architekturmuster werden Objekte nicht als zusammenhängende Datenblöcke im Speicher abgelegt. Stattdessen werden ihre einzelnen Eigenschaften in voneinander getrennten, parallelen Arrays gespeichert. Dies verbessert die Cache-Lokalität, da die \glslink{cpu}{CPU} beim Durchsuchen einer bestimmten Eigenschaft (z.\,B. aller Hash-Werte) diese als kompakten Block am Stück aus dem Arbeitsspeicher (\gls{ram}) in den schnellen \glslink{cpu}{CPU}-Cache laden kann \autocite[161]{fabian2018data}.
\end{definition}

\begin{listing}[ht]
    \caption{Speicherarchitektur im \glslink{soa_acr}{SoA}-Format}
    \label{lst:rs-soa-layout}
    \begin{minted}{typescript}
class RecursiveSet<T extends Value> implements Structural, Iterable<T> {
    // Dichte Arrays (Nutzdaten liegen ohne Lücken nebeneinander)
    private _values: T[] = [];
    private _hashes: number[] = [];
    
    // Indextabelle für die Suchstruktur (Open Addressing)
    private _indices: Uint32Array;
    
    private _bucketCount: number;
    private _mask: number;
}
    \end{minted}
\end{listing}

Um die Effizienz dieser Architektur zu verstehen, müssen die fünf Felder der Klasse getrennt betrachtet werden. Sie teilen sich in zwei logische Ebenen auf: den eigentlichen Datenbestand (die dichten Arrays) und die Navigationsstruktur (die Indextabelle).

\paragraph{Ebene 1: Der dichte Datenbestand}
Die Arrays \mintinline{typescript}{_values} und \mintinline{typescript}{_hashes} wachsen dicht (ohne Lücken) heran. Wenn die Menge drei Elemente enthält, sind exakt die Indizes 0, 1 und 2 belegt. 
\begin{itemize}
    \item \textbf{\texttt{\_values}:} Speichert die tatsächlichen, typisierten Elemente der Menge.
    \item \textbf{\texttt{\_hashes}:} Speichert die zugehörigen, vorberechneten 32-Bit-Hash-Werte. Das Element an Index 5 in \mintinline{typescript}{_values} gehört exakt zum Hash an Index 5 in \mintinline{typescript}{_hashes}. Dass die Hashes in einem eigenen Array liegen, beschleunigt die Suche, da die \glslink{cpu}{CPU} Hash-Werte auf Kollisionen scannen kann, ohne das speicherintensive \mintinline{typescript}{_values}-Array laden zu müssen.
\end{itemize}

% !TEX root = ../master.tex

\paragraph{Ebene 2: Die Navigationsstruktur}
Während die dichten Arrays lediglich die Daten bunkern, kümmert sich die zweite Ebene darum, diese Daten über Hash-Werte effizient wiederzufinden:
\begin{itemize}
    \item \textbf{\texttt{\_indices}:} Dies ist die eigentliche Hashtabelle (die Suchstruktur). Sie ist als \mintinline{typescript}{Uint32Array} implementiert und funktioniert wie ein Inhaltsverzeichnis. Bemerkenswert ist hierbei, dass sie \textbf{keine echten Elemente} speichert, sondern lediglich \textbf{Verweise auf Positionen} in den dichten Arrays. 
    
    Da ein neu angelegtes \mintinline{typescript}{Uint32Array} im Arbeitsspeicher standardmäßig mit Nullen gefüllt ist, nutzt die Bibliothek den Wert \texttt{0} als zweckmäßige Markierung für einen \enquote{leeren Slot}. Um nun den ersten echten Index (die \texttt{0}) der dichten Arrays davon unterscheiden zu können, werden alle gespeicherten Positionen intern um den Wert \texttt{1} erhöht. Ein Eintrag von \texttt{3} in der Indextabelle bedeutet also: \enquote{Ein \emph{möglicher} Treffer für diesen Hash liegt in den dichten Arrays an Position 2}. Ob es sich dabei wirklich um das gesuchte Element oder nur um eine Kollision handelt, und wie der Algorithmus weiter navigiert, wird im Abschnitt~\ref{subsec:rs-intern-kollisionen} (\textit{\glslink{linearprobing}{Linear Probing}}) detailliert beleuchtet.
    
    \item \textbf{\texttt{\_bucketCount}:} Definiert die Gesamtkapazität der \mintinline{typescript}{_indices}-Tabelle (die Anzahl der verfügbaren Such-Slots). Diese Kapazität ist zwingend immer eine Zweierpotenz (z.\,B. 16, 32, 64).
    
    \item \textbf{\texttt{\_mask}:} Die Bitmaske ist ein klassischer Performance-Trick aus der Hardware-Programmierung. Um einen beliebig großen Hash-Wert auf die tatsächliche Größe der Tabelle herunterzubrechen, nutzt man klassischerweise die Modulo-Division (\mintinline{typescript}{hash % _bucketCount}). Da Divisionen für die \glslink{cpu}{CPU} jedoch vergleichsweise teuer sind, wird die Bitmaske auf \mintinline{typescript}{_bucketCount - 1} gesetzt (bei 16 Buckets ist die Maske 15, binär \texttt{00001111}). Die Operation \mintinline{typescript}{hash & _mask} (bitweises UND) liefert exakt dasselbe Ergebnis wie Modulo, wird von der \glslink{cpu}{CPU} als elementare Bit-Operation aber typischerweise in einem einzigen Taktzyklus berechnet.
\end{itemize}

\paragraph{Konzepte der Gleichheit und Type Narrowing}
\label{subsec:rs-intern-areequal}

Wie in Abschnitt~\ref{subsec:rs-intern-hash-engine} beschrieben, dient der Hash-Wert lediglich als Wegweiser, um die Suche in der Hashtabelle einzugrenzen. Da zwei inhaltlich unterschiedliche Mengen rein rechnerisch denselben Hash-Wert generieren können (Kollision), muss die Datenstruktur gefundene Kandidaten iterativ auf Gleichheit prüfen. Hierbei ist eine präzise terminologische Trennung essenziell:

\begin{itemize}
    \item \textbf{Referenzidentität:} Zwei Variablen verweisen auf exakt denselben Bereich im Arbeitsspeicher (\mintinline{typescript}{a === b}).
    \item \textbf{Hash-Gleichheit:} Zwei Objekte generieren denselben numerischen Hash-Wert (\mintinline{typescript}{a.hashCode === b.hashCode}). Dies ist eine notwendige, aber keine hinreichende Bedingung für weitere Gleichheitsannahmen.
    \item \textbf{Strukturelle Gleichheit:} Zwei getrennte Objekte weisen denselben inhaltlichen Aufbau auf (geprüft über die Methode \mintinline{typescript}{equals}).
    \item \textbf{Mathematische Gleichheit:} Die semantische Eigenschaft, dass zwei Mengen dieselben Elemente beinhalten, unabhängig von der Reihenfolge, in der sie im Speicher abgelegt wurden.
\end{itemize}

Genau diese Aufgabe übernimmt die vertraglich im \mintinline{typescript}{Structural}-Interface geforderte \mintinline{typescript}{equals}-Methode. Am Beispiel der Klasse \mintinline{typescript}{RecursiveSet} zeigt sich, wie TypeScript-spezifische Typsicherheit und Performance-Optimierungen hier Hand in Hand gehen:

\begin{listing}[ht]
    \caption{Die \texttt{equals}-Methode von \texttt{RecursiveSet}}
    \label{lst:rs-equals-method}
    \begin{minted}{typescript}
    equals(other: unknown): boolean {
        // 1. Identitäts-Kurzschluss (Referenzidentität)
        if (this === other) return true;
        
        // 2. Typprüfung und Type Narrowing
        if (!(other instanceof RecursiveSet)) return false;
        
        // 3. O(1) Performance-Schranken (Größe und Hash-Gleichheit)
        if (this.size !== other.size) return false;
        if (this.hashCode !== other.hashCode) return false;
        
        // 4. Tiefe strukturelle und mathematische Prüfung
        for (const v of this._values) { 
            if (!other.has(v)) return false; 
        }
        
        return true;
    }
    \end{minted}
\end{listing}

Dieser Algorithmus ist wie ein Trichter aufgebaut: Er versucht, eine aufwendige Tiefenprüfung (Zeile 13–15) durch immer strengere, aber effiziente Vorab-Ausschlusskriterien zu vermeiden (\textit{Short-Circuit Evaluation}).

\paragraph*{Sicherheit durch \texttt{unknown} und \glslink{typenarrowing}{Type Narrowing} (Zeile 1–6)}
Die Signatur der Methode fordert bewusst den Typ \mintinline{typescript}{unknown} für den Parameter \mintinline{typescript}{other}. Wie in Definition~\ref{def:ts-unknown} (Kapitel 2) erläutert, zwingt dieser Typ den Compiler dazu, jegliche blinde Interaktion mit dem Objekt zu verbieten. Die Methode weiß an diesem Punkt nicht, ob \mintinline{typescript}{other} eine Zahl, ein String oder ein komplexes Objekt ist.

Zunächst greift ein schneller Abgleich der \textbf{Referenzidentität} (Zeile 3). Zeigen \mintinline{typescript}{this} und \mintinline{typescript}{other} auf exakt dieselbe Speicheradresse (\mintinline{typescript}{===}), ist jede weitere Prüfung obsolet – ein Objekt ist immer gleich zu sich selbst.

Ist dies nicht der Fall, folgt in Zeile 6 der entscheidende Schritt für die Typsicherheit: Die Laufzeitprüfung \mintinline{typescript}{other instanceof RecursiveSet}. Ist das Fremdobjekt gar keine Menge, wird sofort \mintinline{typescript}{false} zurückgegeben. 

Wird dieser Check jedoch passiert, greift das in Definition~\ref{def:ts-type-narrowing} (Kapitel 2) beschriebene \textbf{\glslink{typenarrowing}{Type Narrowing}}. Der TypeScript-Compiler analysiert den Kontrollfluss und weiß ab Zeile 9 mit Sicherheit, dass \mintinline{typescript}{other} eine Instanz von \mintinline{typescript}{RecursiveSet} ist. Ab hier schaltet der Compiler den typsicheren Zugriff auf Eigenschaften wie \mintinline{typescript}{other.size}, \mintinline{typescript}{other.hashCode} und die Methode \mintinline{typescript}{other.has()} frei.

\paragraph*{$\mathcal{O}(1)$ Performance-Schranken (Zeile 8–10)}
Bevor die Methode die Iteration über alle Elemente startet, werden zwei Eigenschaften verglichen, die in konstanter Zeit ($\mathcal{O}(1)$) abrufbar sind:
\begin{enumerate}
    \item \textbf{Größe (\texttt{size}):} Mengen mit unterschiedlich vielen Elementen können per Definition nicht gleich sein.
    \item \textbf{Hash-Gleichheit (\texttt{hashCode}):} Wenn zwei \mintinline{typescript}{RecursiveSet}-Instanzen inhaltlich gleich sind, \emph{müssen} sie zwingend denselben Hash-Wert besitzen. Unterscheiden sich die Hashes, kann die Prüfung sofort abgebrochen werden. Da der Hash-Wert der Menge intern gecacht wird (siehe Abschnitt~\ref{subsec:rs-intern-freeze}), erfordert dieser Vergleich nahezu keine Rechenzeit.
\end{enumerate}

\paragraph*{Strukturelle und mathematische Prüfung über dichte Arrays (Zeile 12–16)}
Erst wenn das Fremdobjekt \emph{sicher} eine Menge gleicher Größe und mit demselben Hash-Wert ist, greift die Methode auf das in Abschnitt~\ref{subsec:rs-intern-speicherlayout} etablierte \glslink{soa_acr}{SoA}-Layout zurück. Die \mintinline{typescript}{for}-Schleife iteriert über das dichte \mintinline{typescript}{_values}-Array und prüft mittels \mintinline{typescript}{other.has(v)}, ob jedes Element der aktuellen Menge auch in der anderen Menge existiert. 

Da die Methode \mintinline{typescript}{has()} selbst im Durchschnitt \glslink{amortisiert}{amortisiert} in $\mathcal{O}(1)$ arbeitet (unter Annahme einer gleichmäßigen Hash-Verteilung), bleibt auch die gesamte \textbf{strukturelle und mathematische Gleichheitsprüfung} zweier Mengen im Normalfall sehr effizient und steigt nur im Worst-Case auf einen Aufwand von $\mathcal{O}(N)$.

\subsection{Kollisionsauflösung und Index-Verwaltung}
\label{subsec:rs-intern-kollisionen}

Da der Hash-Wert zur Indexberechnung genutzt wird, lässt es sich mathematisch nicht vermeiden, dass zwei unterschiedliche Werte denselben Index in der \mintinline{typescript}{_indices}-Tabelle berechnen (das sogenannte Taubenschlagprinzip). Die Datenstruktur muss diese Hash-Kollisionen auflösen.

\subsubsection{Open Addressing mit Linear Probing}
\label{subsubsec:rs-intern-linear-probing}

Um die Cache-Lokalität des \glslink{soa_acr}{SoA}-Layouts beizubehalten, wird auf externe Zeigerstrukturen verzichtet und stattdessen eine flache Tabellenverwaltung genutzt:

\begin{definition}
\label{def:open-addressing}
\textbf{Open Addressing mit \glslink{linearprobing}{Linear Probing} (Lineares Sondieren):} Bei diesem Verfahren werden alle Verweise direkt im Array der Hashtabelle gespeichert. Berechnet der Hash einen Index (Slot), der bereits von einem anderen Element belegt ist (eine Kollision), rückt der Algorithmus schrittweise zum nächstmöglichen benachbarten Slot weiter, bis er eine Lücke findet oder den gesuchten Wert entdeckt \autocite[293f]{cormen2022introduction}.
\end{definition}

Man kann sich dies wie das Einsortieren von Büchern in einem langen Regal vorstellen: Berechnet das System für ein neues Buch einen Stammplatz, der bereits belegt ist, wird das Buch nicht in einem externen Anbau ausgelagert. Stattdessen prüft man einfach den direkten Nachbarplatz auf der rechten Seite, bis man auf eine Lücke trifft. 

Die Methode \mintinline{typescript}{has} aus Listing~\ref{lst:rs-linear-probing} demonstriert diese Suchlogik im Zusammenspiel mit dem zuvor eingeführten \glslink{soa_acr}{SoA}-Speicherlayout.

\begin{listing}[ht]
    \caption{\glslink{linearprobing}{Linear Probing} am Beispiel der \texttt{has}-Methode}
    \label{lst:rs-linear-probing}
    \begin{minted}{typescript}
has(element: T): boolean {
    const h = getHashCode(element);
    let idx = h & this._mask; // Start-Slot berechnen
    
    while (true) {
        const entry = this._indices[idx];
        if (entry === 0) return false; // Lücke -> Element fehlt
        
        const valIndex = entry - 1; // 1-basierten Verweis umrechnen
        
        // 1. Hash vergleichen (schnell), 2. equals vergleichen (teuer)
        if (this._hashes[valIndex] === h && 
            areEqual(this._values[valIndex], element)) {
            return true; // Treffer
        }
        
        // Kollision -> Gehe zum nächsten Slot (mit Wrap-around)
        idx = (idx + 1) & this._mask; 
    }
}
    \end{minted}
\end{listing}

Dieser Suchalgorithmus ist der Programmabschnitt, der zur Laufzeit am häufigsten ausgeführt wird. Da diese Suchlogik die Basis für nahezu jede Interaktion mit der Hashtabelle bildet, sei es beim Hinzufügen (\mintinline{typescript}{add}), Entfernen (\mintinline{typescript}{remove}) oder beim reinen Nachschlagen (\mintinline{typescript}{has}), ist sie auf Effizienz ausgelegt. Sie kommt (mit Ausnahme des Aufrufs von \mintinline{typescript}{areEqual} bei einem Kandidaten) ohne weitere Funktionsaufrufe aus. Der Ablauf lässt sich in vier Schritte zerlegen:

\paragraph*{Schritt 1: Den Start-Slot berechnen (Zeile 2–3)}
Zunächst wird der 32-Bit-Hash des gesuchten Elements berechnet. Da dieser Wert weit außerhalb der Tabellengröße liegen kann, wird er mittels der Bitmaske (\mintinline{typescript}{h & this._mask}) auf einen gültigen Start-Index (\mintinline{typescript}{idx}) in der \mintinline{typescript}{_indices}-Tabelle heruntergebrochen.

\paragraph*{Schritt 2: Die Abbruchbedingung der Schleife (Zeile 5–7)}
Die \mintinline{typescript}{while (true)}-Schleife wirkt potenziell endlos. Hier greift jedoch eine Eigenschaft der Hashtabelle: Die Bibliothek stellt sicher, dass die Tabelle niemals zu mehr als $75\,\%$ gefüllt ist (der sogenannte Load-Factor, siehe Abschnitt~\ref{subsec:rs-intern-resize-shrink}). Es existiert also \emph{immer} mindestens ein leerer Slot. 

Wird ein Slot mit dem Wert \texttt{0} (leer) gefunden, stoppt die Suche sofort und liefert \mintinline{typescript}{false}. Warum? Weil das Einsortieren im Regal (\textit{\glslink{linearprobing}{Linear Probing}}) garantiert, dass bei einer Kollision alle betroffenen Elemente nahtlos hintereinander abgelegt werden. Eine Lücke bedeutet zwingend, dass der Suchpfad an dieser Stelle endet und das gesuchte Element folglich nicht in der Menge existiert.

\paragraph*{Schritt 3: Dereferenzierung und Hash-Vorabgleich (Zeile 9–15)}
Ist der Slot belegt (\mintinline{typescript}{entry > 0}), wird der 1-basierte Verweis durch \mintinline{typescript}{entry - 1} in den korrekten 0-basierten Index (\mintinline{typescript}{valIndex}) für die dichten Nutzdaten-Arrays umgerechnet. 

Nun zeigt sich ein Vorteil des \glslink{soa_acr}{SoA}-Layouts: Anstatt sofort die \mintinline{typescript}{areEqual}-Funktion aufzurufen, wird zunächst der gespeicherte Hash-Wert (\mintinline{typescript}{this._hashes[valIndex]}) mit dem gesuchten Hash (\mintinline{typescript}{h}) verglichen. Stimmen die Hashes nicht überein, liegt eine klassische Kollision vor und die Inhaltsprüfung wird übersprungen. Nur wenn die Hashes identisch sind (\textbf{Hash-Gleichheit}), wird \mintinline{typescript}{areEqual} zur Verifizierung der \textbf{strukturellen Gleichheit} aufgerufen.

\paragraph*{Schritt 4: Linear Probing und der Wrap-around (Zeile 18)}
Stellte sich der belegte Slot als Kollision heraus, muss der Algorithmus den nächsten benachbarten Slot prüfen. Die Anweisung \mintinline{typescript}{idx = (idx + 1) & this._mask} inkrementiert den Index nicht nur um $1$, sondern sorgt gleichzeitig für den sogenannten \textbf{Wrap-around}. Erreicht der Suchlauf das physische Ende des Arrays (das Ende des Regals), setzt das bitweise UND (\mintinline{typescript}{& this._mask}) den Wert vollautomatisch auf \texttt{0} zurück. Der Algorithmus springt also an den Anfang der Tabelle und setzt die Suche dort fort, ohne dass \mintinline{typescript}{if}-Abfragen für Array-Grenzen benötigt werden.

\subsubsection{Löschvorgang: Tombstones und Backshift Deletion}
\label{subsubsec:rs-intern-backshift}

Das Entfernen von Elementen aus einer Hashtabelle mit \textit{Open Addressing} ist komplexer als bei verketteten Listen. Nimmt man ein Element einfach aus seinem Slot im Regal (durch Zurücksetzen auf \texttt{0}), entsteht eine harte Lücke. Dies hat Folgen für das \textit{\glslink{linearprobing}{Linear Probing}}: Suchen wir nach einem Element, das aufgrund früherer Kollisionen hinter dieser neuen Lücke einsortiert wurde, bricht die \mintinline{typescript}{while}-Schleife an der leeren Stelle ab und meldet, das Element existiere nicht. Die Sondierungskette ist gerissen.

Der klassische Lösungsansatz für dieses Problem ist die Verwendung sogenannter \glslink{tombstone}{Tombstones}:

\begin{definition}
\label{def:tombstones}
\textbf{\glslink{tombstone}{Tombstones} (Grabsteine):} Anstatt einen Slot im Regal wirklich zu leeren (\texttt{0}), wird ein spezielles Marker-Objekt (der \glslink{tombstone}{Tombstone}) an dieser Stelle platziert. Dieser Marker signalisiert dem Suchalgorithmus: \enquote{Hier wurde zwar etwas gelöscht, aber brich die Suche nicht ab, sondern suche im nächsten Slot weiter.}
\end{definition}

Obwohl \glslink{tombstone}{Tombstones} einfach zu implementieren sind, führen sie bei vielen Einfüge- und Löschvorgängen dazu, dass die Tabelle über die Zeit mit Grabsteinen \enquote{verschmutzt}. Die Suchwege werden künstlich in die Länge gezogen, was die Zugriffszeiten verschlechtert. Zudem müssen \glslink{tombstone}{Tombstones} regelmäßig durch vollständige Neu-Aufbauten der Tabelle (Rehashing) entfernt werden.

\paragraph*{Die Lösung: Backshift Deletion}
Um dies zu vermeiden, nutzt die Bibliothek die \textit{\glslink{backshift}{Backshift Deletion}} (Rückwärts-Verschiebung) – eine Technik, die auf dem \textit{Robin Hood Hashing} aufbaut \autocite{celis1986robin}. 

Wird ein Element gelöscht, entsteht zunächst eine Lücke. Der Algorithmus, siehe Listing~\ref{lst:rs-backshift-deletion}, blickt nun in die nachfolgenden Slots und prüft, ob Elemente weiter hinten im Regal stehen, die eigentlich weiter vorne (also näher an der Lücke oder genau darin) stehen müssten. Ist das der Fall, rücken diese Elemente auf, und die Lücke wandert weiter nach hinten, bis sie das Ende der Kollisionskette erreicht und geleert werden kann.

% !TEX root = ../master.tex

\begin{listing}[H]
    \caption{Reparatur der Sondierungskette (Backshifting)}
    \label{lst:rs-backshift-deletion}
    \begin{minted}{typescript}
private removeIndex(holeIdx: number): void {
    let i = (holeIdx + 1) & this._mask;
    
    while (this._indices[i] !== 0) {
        const entry = this._indices[i];
        const valPtr = entry - 1;
        const h = this._hashes[valPtr];
        
        const idealIdx = h & this._mask;
        
        const distToHole = (holeIdx - idealIdx + this._bucketCount) & this._mask;
        const distToI = (i - idealIdx + this._bucketCount) & this._mask;
        
        if (distToHole < distToI) {
            this._indices[holeIdx] = entry;
            holeIdx = i;
        }
        i = (i + 1) & this._mask;
    }
    this._indices[holeIdx] = 0;
}
    \end{minted}
\end{listing}

Dieser Algorithmus ist ein typisches Beispiel für hardwarenahe Optimierung in TypeScript. Da er ohne Funktionsaufrufe auskommt, übernimmt die Bit-Mathematik die Steuerung. Der Ablauf lässt sich in fünf Phasen unterteilen:

\begin{enumerate}
    \item \textbf{Start der Sondierung (Zeile 2):} Der Algorithmus startet nicht bei der Lücke (\mintinline{typescript}{holeIdx}), sondern prüft direkt den ersten rechten Nachbarn. Das bitweise UND (\mintinline{typescript}{& this._mask}) sichert auch hier den zirkulären Wrap-around ab, falls die Lücke am äußersten Ende des Arrays lag.
    
    \item \textbf{Abbruchbedingung und Dereferenzierung (Zeile 4–7):} Die Schleife iteriert, bis sie auf einen nativ leeren Slot (\texttt{0}). Ist ein Slot belegt, wird der 1-basierte Verweis (\mintinline{typescript}{entry}) wieder in den 0-basierten Index (\mintinline{typescript}{valPtr}) für das dichte \mintinline{typescript}{_hashes}-Array umgerechnet, um den Hash-Wert des aktuellen Elements auszulesen.
    
    \item \textbf{Ideal-Slot und zirkuläre Distanz-Metrik (Zeile 9–13):} In Zeile 9 wird berechnet, an welchem Index das Element ursprünglich hätte landen sollen (\mintinline{typescript}{idealIdx}). Danach folgt der mathematische Kern: Wie weit ist das Element aktuell von seinem Ideal-Slot entfernt (\mintinline{typescript}{distToI}) und wie weit wäre es entfernt, wenn es in die Lücke rutschen würde (\mintinline{typescript}{distToHole})? 
    
    Die Addition von \mintinline{typescript}{this._bucketCount} ist hier essenziell: Da das Array zirkulär ist, könnte \mintinline{typescript}{holeIdx - idealIdx} negativ werden. In JavaScript würde der Modulo-Operator bei negativen Zahlen ungeeignete Ergebnisse liefern. Die Addition stellt sicher, dass der Wert positiv bleibt, bevor die Bitmaske ihn wieder in den zirkulären Ring umrechnet.
    
    \item \textbf{Die Robin-Hood-Bedingung (Zeile 15–18):} Nun greift die eigentliche \textit{Backshift}-Logik. Ist die Distanz zur Lücke kleiner als die aktuelle Distanz, also gilt \mintinline{typescript}{distToHole < distToI}, profitiert das Element von einem Umzug. Es wird physisch in die Lücke verschoben (Zeile 16). Die Lücke wandert dadurch auf die alte Position des Elements (\mintinline{typescript}{holeIdx = i}) und wird im nächsten Schleifendurchlauf weiter nach hinten gereicht.
    
    \item \textbf{Abschluss (Zeile 21):} Sobald die Schleife auf einen nativ leeren Slot trifft, ist die Kette repariert. Die nach hinten durchgereichte Lücke befindet sich nun am Ende der Sondierungskette und kann mit \texttt{0} (leer) überschrieben werden.
\end{enumerate}

\paragraph*{Nutzdaten entfernen: Der Dense-Swap}
Nachdem die \mintinline{typescript}{removeIndex}-Methode die Sondierungskette repariert hat, muss das physische Element aus den dichten Nutzdaten-Arrays (\mintinline{typescript}{_values} und \mintinline{typescript}{_hashes}) gelöscht werden, ohne Lücken zu hinterlassen.

Ein nativer Aufruf wie \mintinline{typescript}{this._values.splice()} würde das Element zwar entfernen, müsste danach aber alle nachfolgenden Elemente im Arbeitsspeicher um einen Platz nach vorne rücken. Dies würde zu einer linearen Laufzeit von $\mathcal{O}(N)$ führen.

Stattdessen nutzt die Bibliothek den \textbf{Dense-Swap} (Tausch im dichten Array). Das zu löschende Element wird mit dem \textit{allerletzten} Element des Arrays überschrieben. Danach wird das Array mittels \mintinline{typescript}{pop()} in konstanter Zeit ($\mathcal{O}(1)$) am Ende um einen Platz verkürzt. Da sich durch diesen Tausch die physische Position des letzten Elements geändert hat, muss abschließend nur noch der zugehörige Verweis in der Indextabelle auf den neuen Index aktualisiert werden. 

Durch diese Kombination aus \textit{\glslink{backshift}{Backshift Deletion}} und \textit{Dense-Swap} garantiert die Bibliothek eine stabile Performance bei Löschvorgängen.

\subsection{Dynamische Kapazitätsverwaltung (Resizing und Rehashing)}
\label{subsec:rs-intern-resize-shrink}

Damit das \textit{\glslink{linearprobing}{Linear Probing}} (Abschnitt~\ref{subsubsec:rs-intern-linear-probing}) performant bleibt und die \mintinline{typescript}{while}-Schleife nicht endlos läuft, muss ein gewisser Anteil der Indextabelle leer bleiben. Je voller die Tabelle wird, desto wahrscheinlicher werden Kollisionen und desto länger werden die Sondierungsketten.

Gesteuert wird dies über den sogenannten \textit{Load Factor}:

\begin{definition}
\label{def:load-factor}
\textbf{Load Factor (Füllgrad):} Der Load Factor beschreibt das Verhältnis von gespeicherten Elementen zur Gesamtkapazität der Hashtabelle ($N / \text{Capacity}$). Die Bibliothek definiert eine feste Obergrenze von $0.75$ ($75\,\%$). Sobald ein neues Element diesen Schwellenwert überschreitet, verdoppelt die Tabelle ihre Kapazität. 
\end{definition}

Gleichzeitig arbeitet die Bibliothek speicherschonend: Fällt der Füllgrad durch Löschvorgänge unter $0.25$ ($25\,\%$), wird die Tabellengröße halbiert (\textit{Shrinking}), sofern eine Mindestgröße (z.\,B. 16 Slots) nicht unterschritten wird. Da die Kapazität immer verdoppelt oder halbiert wird, bleibt sie stets eine strikte Zweierpotenz, was die Voraussetzung für den Einsatz der Bitmaske (\mintinline{typescript}{_mask}) ist.

\paragraph*{Das Rehashing: Ein Umzug ohne Inhaltskontrolle}
Wenn die Tabelle wächst oder schrumpft, können die alten Indizes nicht direkt kopiert werden. Da sich die Kapazität (\mintinline{typescript}{_bucketCount}) ändert, ändert sich unweigerlich auch die Bitmaske (\mintinline{typescript}{_mask}). Ein Hash-Wert, der zuvor auf Slot $3$ abgebildet wurde, könnte in der neuen Tabelle auf Slot $19$ verweisen. 

Die Suchstruktur muss daher neu berechnet werden (Rehashing):

\begin{listing}[ht]
    \caption{Der Rehashing-Prozess bei einer Kapazitätsänderung}
    \label{lst:rs-rehashing}
    \begin{minted}{typescript}
    private resize(newCapacity: number): void {
        this._bucketCount = newCapacity;
        this._mask = newCapacity - 1;
        
        // Frische Indextabelle anlegen (standardmäßig mit Nullen gefüllt)
        this._indices = new Uint32Array(newCapacity);

        // Alle existierenden Elemente neu einsortieren
        for (let i = 0; i < this._values.length; i++) {
            const h = this._hashes[i];
            let idx = h & this._mask; // Neuen Ideal-Slot berechnen

            // Vereinfachtes Linear Probing ohne equals-Prüfung
            while (this._indices[idx] !== 0) {
                idx = (idx + 1) & this._mask; 
            }
            
            this._indices[idx] = i + 1; // 1-basierten Verweis speichern
        }
    }
    \end{minted}
\end{listing}

Dieser Algorithmus verdeutlicht einen architektonischen Vorteil des \glslink{soa_acr}{SoA}-Lay\-outs. Da die Nutzdaten (\mintinline{typescript}{_values} und \mintinline{typescript}{_hashes}) als dicht gepackte Arrays vorliegen, müssen sie beim Rehashing nicht verschoben werden. Der Umzug betrifft ausschließlich die flache \mintinline{typescript}{_indices}-Tabelle.

Zudem profitiert der Code von einem signifikanten Performance-Vorteil: Da das \mintinline{typescript}{_values}-Array per Definition bereits bereinigt ist und ausschließlich eindeutige Elemente enthält, kann der Algorithmus die \mintinline{typescript}{areEqual}-Aufrufkette vollständig ignorieren. Er prüft bei einer Kollision lediglich, ob der Slot leer ist (\mintinline{typescript}{!== 0}), und rückt weiter.

\paragraph*{Amortisierte Laufzeit}
Auf den ersten Blick verletzt das Rehashing das Versprechen einer Hashtabelle, Elemente im Durchschnitt in $\mathcal{O}(1)$ einzufügen. Muss die Tabelle wachsen, erfordert die \mintinline{typescript}{for}-Schleife einen Aufwand von $\mathcal{O}(N)$, da alle $N$ Elemente neu einsortiert werden.

In der Informatik wird dieses Verhalten über die \textit{\glslink{amortisiert}{amortisierte Laufzeitanalyse}} bewertet:

\begin{definition}
\label{def:amortisierte-laufzeit}
\textbf{\glslink{amortisiert}{Amortisierte Laufzeit}:} Dieser Begriff beschreibt den durchschnittlichen Aufwand einer Operation über eine sehr lange Sequenz von Aufrufen. Da die Tabelle ihre Größe jedes Mal verdoppelt, passieren Resizing-Operationen seltener. Die vereinzelten $\mathcal{O}(N)$-Aufwände \enquote{amortisieren} (verteilen) sich auf die weitaus zahlreicheren, sehr effizienten $\mathcal{O}(1)$-Einfügungen. Im Durchschnitt bleibt der Aufwand für eine einzelne Einfügung somit bei \glslink{amortisiert}{amortisiert} $\mathcal{O}(1)$.
\end{definition}

Die Kombination aus Verdoppeln beim Wachsen, Halbieren beim Schrumpfen und Verzicht auf Inhalts\-prüfungen während des Rehashings garantiert architektonisch ein Gleichgewicht zwischen Speicherhaushalt und Zugriffsgeschwindigkeit.

\subsection{Technische Umsetzung des Freeze-Contracts}
\label{subsec:rs-intern-freeze}

In Abschnitt~\ref{subsubsec:rs-api-freeze} wurde der \emph{Freeze-Contract} aus \glslink{api}{API}-Sicht eingeführt. Dieser besagt, dass Mengen aus Sicherheitsgründen unveränderlich werden, sobald sie in eine andere Struktur geschachtelt werden. Technisch wird dieser Mechanismus über die Abfrage des Hash-Werts gesteuert.

Da \mintinline{typescript}{RecursiveSet} eine ungeordnete Menge repräsentiert, darf der Hash-Wert nicht von der historischen Einfügereihenfolge abhängen. Dies wird durch eine fortlaufende XOR-Verknüpfung (\mintinline{typescript}{_xorHash ^= h;}) beim Einfügen sichergestellt, da die XOR-Operation kommutativ ist.

Der tatsächliche Freeze-Vorgang wird ausgelöst, sobald eine übergeordnete Struktur den Getter \mintinline{typescript}{get hashCode()} der Kind-Menge aufruft:

\begin{listing}[ht]
    \caption{Einfrieren durch Hash-Abruf}
    \label{lst:rs-freeze-implementation}
    \begin{minted}{typescript}
    get hashCode(): number {
        if (this._isFrozen) return this._xorHash;

        // Iteriert über alle Elemente und berührt deren hashCode, 
        // wodurch der Freeze-Effekt nach unten durchgereicht wird.
        for (let i = 0; i < this._values.length; i++) {
            const v = this._values[i];
            if (v && typeof v === 'object') {
                v.hashCode; 
            }
        }

        this._isFrozen = true;
        return this._xorHash;
    }
    \end{minted}
\end{listing}

Sobald \mintinline{typescript}{this._isFrozen} auf \mintinline{typescript}{true} gesetzt ist, blockieren die Methoden \mintinline{typescript}{add} und \mintinline{typescript}{remove} mit einer Fehlermeldung. 

Die \mintinline{typescript}{for}-Schleife ruft proaktiv \mintinline{typescript}{v.hashCode} auf allen gespeicherten Objekten auf. Dies erzeugt einen kaskadierenden Effekt:

\begin{definition}
\label{def:kaskadierung}
\textbf{Kaskadierung (Cascading):} Abgeleitet vom Begriff der Kaskade (einem stufenförmigen Wasserfall) beschreibt dies in der Informatik einen Effekt, der sich automatisch von einer oberen Ebene über alle darunterliegenden Ebenen fortpflanzt. Wenn die Eltern-Menge eingefroren wird, friert sie automatisch alle in ihr enthaltenen Kind-Strukturen ein. Diese frieren wiederum ihre eigenen Kinder ein, bis das Netz der verschachtelten Datenstrukturen versiegelt ist.
\end{definition}

Da der \mintinline{typescript}{hashCode}-Getter bei allen strukturellen Elementen wie \mintinline{typescript}{RecursiveSet}, \mintinline{typescript}{Tuple} oder \mintinline{typescript}{RecursiveMap} exakt denselben Mechanismus beinhaltet, kaskadiert das Einfrieren durch den gesamten Baum hinab.
